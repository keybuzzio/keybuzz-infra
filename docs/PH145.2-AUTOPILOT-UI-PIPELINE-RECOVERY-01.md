## 4. Cause Racine Unique (ETAPE 5)

### Constat factuel

| Question | Reponse | Preuve |
|---|---|---|
| L'API genere-t-elle des drafts ? | OUI | POST /autopilot/evaluate retourne DRAFT_GENERATED |
| Le BFF proxy-t-il correctement ? | OUI | Logs API PROD montrent appels internes |
| Le client fetch-t-il les drafts ? | OUI | Logs API montrent GET /autopilot/draft depuis navigateur |
| Y a-t-il des drafts actifs pour le tenant AUTOPILOT ? | NON | 0 entrees ai_action_log pour switaa-sasu-mnc1ouqu |
| Pourquoi 0 drafts ? | evaluate JAMAIS appele automatiquement | voir ci-dessous |

### Cause racine

**`evaluateAndExecute()` n'est appele QUE depuis `POST /autopilot/evaluate` (routes.ts L179).
Le pipeline inbound (`src/modules/inbound/routes.ts`) n'appelle JAMAIS l'autopilot engine.**

```
ATTENDU (PH143 docs) :
  Message client entrant → inbound/routes.ts → evaluateAndExecute() → DRAFT_GENERATED

REEL :
  Message client entrant → inbound/routes.ts → evaluatePlaybooksForConversation() → FIN
                                                (pas d'appel autopilot)
```

### Preuve supplementaire

- Le pipeline inbound appelle `evaluatePlaybooksForConversation` (fire-and-forget) apres chaque message — mais PAS `evaluateAndExecute`
- Le seul point d'appel de `evaluateAndExecute` dans TOUT le codebase est `routes.ts` ligne 179 (endpoint POST /autopilot/evaluate)
- grep confirm : `evaluateAndExecute` n'est importe/utilise nulle part dans inbound/, messages/, ou tout autre module

### Datation

Ce chaînon manquant n'est PAS une regression PH145. Il n'a probablement JAMAIS ete connecte. Les tests PH142/PH143 ont tous utilise des appels manuels a `POST /autopilot/evaluate`.

## 5. Fix Minimal Applique (ETAPE 6)

### Modification : `src/modules/inbound/routes.ts`

3 changements minimaux :

**1. Import (ligne 2)**
```typescript
import { evaluateAndExecute } from '../autopilot/engine';
```

**2. Endpoint email — apres playbook evaluation (ligne 290-292)**
```typescript
// PH145.2: Evaluate autopilot for this conversation (fire-and-forget)
if (body.tenantId && conversationId) {
  evaluateAndExecute(conversationId, body.tenantId, 'inbound')
    .catch(err => console.error('[Autopilot] Inbound auto-evaluate error:', err.message));
}
```

**3. Endpoint Amazon forward — apres playbook evaluation (ligne 577-579)**
```typescript
// PH145.2: Evaluate autopilot for Amazon inbound (fire-and-forget)
if (inboundPayload.tenantId && conversationId) {
  evaluateAndExecute(conversationId, inboundPayload.tenantId, 'inbound')
    .catch(err => console.error('[Autopilot] Amazon inbound auto-evaluate error:', err.message));
}
```

### Securite du fix

- Fire-and-forget : l'appel autopilot ne bloque PAS la reponse inbound (`.catch()`)
- L'engine verifie internement `resolveIAMode` + `canUseAutopilot` : les tenants sans plan AUTOPILOT sont ignores (MODE_NOT_AUTOPILOT)
- Les tenants avec `safe_mode: true` generent des DRAFT_GENERATED (pas d'envoi auto)
- Les guardrails PH145 sont actifs (scoring risque, fausses promesses, etc.)
- Aucun impact sur les tenants PRO/STARTER : l'engine retourne `noop` immediatement
