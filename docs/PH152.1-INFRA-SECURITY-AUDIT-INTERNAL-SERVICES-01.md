### Erreur TLS expliquee

Les services utilisaient historiquement `https://vault.keybuzz.io:8200` avec un helper `vaultFetch` (bypass hostname verification). Mais Vault a toujours eu `tls_disable = 1`. L'erreur `wrong version number` vient du fait que `curl -sk https://...` envoie un ClientHello TLS a un serveur HTTP pur.

### Acces reel depuis K8s

Le service K8s `vault` (namespace `default`, ClusterIP `10.111.0.31`) pointe vers les endpoints :

- `10.0.0.150:8200`
- `10.0.0.154:8200`
- `10.0.0.155:8200`

Test depuis un pod K8s : `http://vault.default.svc.cluster.local:8200/v1/sys/health` → **OK** (initialized, unsealed).

L'env var API PROD est correcte : `VAULT_ADDR=http://vault.default.svc.cluster.local:8200` (HTTP, pas HTTPS).

### Conclusion sur le helper `vaultFetch`

Le helper `vaultFetch` avec `checkServerIdentity: () => undefined` dans le code backend est un **vestige** de l'epoque ou les services essayaient d'acceder a Vault via HTTPS. Vault n'a jamais eu TLS active. Les services qui utilisent `VAULT_ADDR=http://...` via l'env K8s fonctionnent correctement sans ce helper.

## 5. RISQUES

### Risques identifies


| #   | Risque                                        | Niveau | Type      | Detail                                                                                                                                                                                                                                                |
| --- | --------------------------------------------- | ------ | --------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| R1  | Vault ecoute en HTTP sans TLS                 | D2     | Theorique | Vault est accessible uniquement via le reseau prive Hetzner (10.0.0.0/16). Le trafic interne n'est pas chiffre, mais il reste dans le datacenter prive. Risque reel uniquement si un attaquant a deja acces au reseau prive.                          |
| R2  | Vault bind sur 0.0.0.0                        | D2     | Theorique | Vault ecoute sur toutes les interfaces (y compris eth0 publique). Protege par le firewall Hetzner Cloud qui bloque le port 8200 depuis Internet. Non accessible en test.                                                                              |
| R3  | MinIO bind sur `*:9000` et `*:9001`           | D2     | Theorique | Meme situation que Vault : bind sur toutes interfaces, protege par le reseau Hetzner. Non accessible depuis Internet.                                                                                                                                 |
| R4  | iptables vides sur tous les serveurs internes | D1     | Reel      | Tous les serveurs ont une politique INPUT=ACCEPT sans regles de filtrage. La securite repose entierement sur le firewall Hetzner Cloud et l'isolation reseau. Si le firewall Hetzner est mal configure ou desactive, tous les ports seraient exposes. |
| R5  | Ingress Studio exposes (13j)                  | D1     | Reel      | `studio-dev.keybuzz.io`, `studio.keybuzz.io`, `studio-api-dev.keybuzz.io`, `studio-api.keybuzz.io` sont exposes via Ingress sans mention d'auth. Le Studio a ete retire du build client mais les Ingress et deployments K8s existent toujours.        |
| R6  | `VAULT_TOKEN` = root token                    | D1     | Reel      | L'env `VAULT_TOKEN` reference le secret `vault-root-token` — c'est le token root Vault. En production, il est recommande d'utiliser des tokens limites (AppRole ou Token policies).                                                                   |

## 6. RESUME

### Points positifs

1. **Aucun service de donnees expose sur Internet** (PostgreSQL, Redis, RabbitMQ, MinIO)
2. **Vault est fonctionnel** et activement utilise (30 ExternalSecrets synchronises)
3. **Le reseau prive Hetzner** protege efficacement les services internes
4. **Patroni est complet** (1 leader + 2 replicas, lag=0)
5. **Redis HA restaure** (2 replicas connectes)
6. **Tous les services web ont une forme d'authentification**

### Points d'attention

1. **iptables vides** — la securite repose uniquement sur Hetzner Cloud
2. **Vault en HTTP pur** — acceptable dans un reseau prive mais pas ideal
3. **Vault root token en production** — devrait etre remplace par des tokens AppRole
4. **Ingress Studio orphelins** — a supprimer si le service n'est plus utilise
5. **Vault bind 0.0.0.0 + MinIO bind `*`** — devrait etre restreint a l'IP privee
