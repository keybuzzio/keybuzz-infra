# PH28.12 - Fix "check-email" /login (users existants non reconnus)

**Date:** 2026-02-02  
**Status:** ✅ COMPLETED  
**Environments:** DEV + PROD

---

## 1. Problème Identifié

### Symptôme
Sur `/login`, même pour des emails existants, le système affichait "Aucun compte trouvé" au lieu de proposer l'OTP/SSO.

### Cause Racine
1. L'endpoint `/api/auth/check-email` (client) appelait `/tenant-context/me` (backend) qui **auto-crée les utilisateurs** s'ils n'existent pas
2. L'endpoint `/tenant-context/tenants` utilisait également `getUserFromEmail()` qui auto-crée les utilisateurs
3. Le résultat: tout email passait par le flow OTP car il était créé automatiquement lors du check

---

## 2. Corrections Apportées

### Backend (`keybuzz-api/src/modules/auth/tenant-context-routes.ts`)

#### A. Nouvel endpoint READ-ONLY `/tenant-context/check-user`
```typescript
app.get('/check-user', async (request, reply) => {
  const email = getEmailFromHeader(request);
  
  // Only SELECT - do NOT create
  const userResult = await pool.query(
    'SELECT id, email, name FROM users WHERE LOWER(email) = LOWER($1)',
    [email.trim()]
  );
  
  if (userResult.rows.length === 0) {
    return reply.send({ exists: false, hasTenants: false });
  }
  
  // Get tenant count
  const tenantsResult = await pool.query(
    'SELECT COUNT(*) FROM user_tenants WHERE user_id = $1',
    [user.id]
  );
  
  return reply.send({
    exists: true,
    hasTenants: parseInt(tenantsResult.rows[0].count) > 0,
  });
});
```

#### B. Nouvelle fonction `getUserFromEmailReadOnly()`
- Ne crée JAMAIS d'utilisateur
- Retourne `null` si l'utilisateur n'existe pas
- Utilisée par `/tenant-context/tenants` pour éviter l'auto-création

#### C. Endpoint `/tenant-context/tenants` modifié
- Utilise `getUserFromEmailReadOnly()` au lieu de `getUserFromEmail()`
- Retourne `{ tenants: [], hasNoTenants: true }` si l'utilisateur n'existe pas

### Frontend (`keybuzz-client/app/api/auth/check-email/route.ts`)

```typescript
// PH28.12: Use check-user endpoint that doesn't auto-create users
const response = await fetch(`${API_BASE}/tenant-context/check-user`, {
  method: 'GET',
  headers: {
    'Content-Type': 'application/json',
    'X-User-Email': email.toLowerCase().trim(),
  },
});

if (response.ok) {
  const data = await response.json();
  return NextResponse.json({ 
    exists: data.exists,
    hasTenants: data.hasTenants,
  });
}
```

## 7. Notes Techniques

### Pourquoi le fix Docker a pris du temps
- Le Docker build utilisait des layers cachés malgré `--no-cache`
- Solution: `npm run build` local d'abord, puis Docker build pour réutiliser le `.next` généré
- `kubectl set image` requis car `kubectl apply` ne mettait pas à jour l'image (bug de synchronisation GitOps)

### Flow de login corrigé
```
1. User entre email sur /login
2. Client appelle /api/auth/check-email
3. check-email appelle /tenant-context/check-user (READ-ONLY)
4. Si exists=false → affiche "Aucun compte trouvé" + CTA signup
5. Si exists=true → envoie OTP via /api/auth/magic/start
6. User entre OTP → signIn() crée session
7. TenantProvider détecte session → appelle /tenant-context/me (auto-create OK à ce stade)
```

---

## 8. Conclusion

Le bug était causé par l'auto-création d'utilisateurs lors de la vérification d'existence. La solution consistait à créer un endpoint READ-ONLY (`/check-user`) qui ne crée jamais d'utilisateurs, permettant au flow de login de correctement distinguer les utilisateurs existants des nouveaux.

**Impact:** Les utilisateurs existants peuvent maintenant se connecter normalement via OTP, tandis que les nouveaux utilisateurs sont redirigés vers le signup.
