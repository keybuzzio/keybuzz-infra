## 4. Blocage identifie — Regression si rebuild partiel

L'image validee a ete construite depuis le bastion sur la branche `release/client-v3.5.220@2adbd40` avec des modifications locales non commitees. Ces modifications ont ete effacees du bastion.

Le workspace local est sur une **branche differente** (`main` → `ph152.2-autopilot-draft-banner`) avec des commits **differents** :

- `e3329606` PH142-O3: SLA inbox + BFF routes
- `79b6ccef` PH145.4: Amazon channel badge fix
- `ccfe0361` PH147: autopilotGuardrails sync
- `ef0a5c1c` PH152.2: AutopilotDraftBanner

**Consequence** : committer un feature isole sur cette branche et rebuilder depuis Git produirait une image sur un code base different, avec risque de regression des features P0 deja presentes dans l'image validee.

Un rebuild securise necessite de committer **TOUS** les fichiers modifies/untracked d'un coup (20 M + 18 U pertinents), ce qui depasse le scope d'une "micro-recuperation unique".

## 8. Risque majeur identifie — Preservation du code source

Le code source des features P0-P1 presentes dans l'image validee existe UNIQUEMENT dans :

1. **L'image Docker compilee** (pas de source lisible)
2. **Le workspace local Windows** (`c:\DEV\KeyBuzz\V3`)

Si le workspace local est perdu ou modifie, le code source de ces features devient **irrecuperable**. Un rebuild depuis Git HEAD (bastion `release/client-v3.5.220@2adbd40`) produirait une image sans les features PH144-PH151.2.

**Recommandation** : une phase dediee de "commit atomique de la verite validee" est necessaire AVANT tout rebuild futur. Cette phase doit :

1. Committer TOUS les fichiers modifies + untracked pertinents en une seule operation
2. Pusher vers GitHub
3. Puller sur le bastion
4. Verifier que le build depuis ce commit produit une image equivalente a l'image validee
