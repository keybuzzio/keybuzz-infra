**Date**: 2026-02-22
**Environnement**: DEV uniquement
**Statut**: DEPLOYE + VALIDE — STOP POINT AVANT PROD

---

## 1. Cause racine (rappel PH36.4)

Le pipeline d'import des discussions Octopia (PH35.x) existait en code source mais **n'était pas compilé ni déployé** dans l'image Docker de l'API.

| Problème | Détail |
|----------|--------|
| Import incorrect dans `app.ts` | `./modules/octopia/routes` (stub) au lieu de `./modules/marketplaces/octopia` |
| Worker non démarré | `startOctopiaSyncWorker()` jamais appelé |
| Code non compilé | `dist/modules/marketplaces/` absent de l'image |

## 2. Correction appliquée

### Ce qui a été fait

Le code source côté serveur (`/opt/keybuzz/keybuzz-api/`) contenait **déjà** les bonnes corrections depuis PH35.3 :

```typescript
// app.ts — DEJA CORRECT sur le serveur
import { startOctopiaSyncWorker } from './workers/octopiaSyncWorker';          // ligne 2
import { octopiaRoutes } from './modules/marketplaces/octopia/octopia.routes'; // ligne 46
await octopiaRoutes(app, { pool });                                            // ligne 178
startOctopiaSyncWorker();                                                      // ligne 190
```

**Le problème était uniquement que l'image Docker n'avait JAMAIS été rebuild depuis ces modifications.**

### Aucune modification de code requise

Zéro modification de fichier source. Le fix est un **rebuild déterministe** de l'image Docker à partir du code source existant.

## 9. Problèmes connus (hors scope PH36.4A)

1. **Inbox conversations vide** : Le client charge les conversations depuis `api.keybuzz.io` (PROD) au lieu de `api-dev.keybuzz.io` (DEV) à cause du cache navigateur du chunk `1024-*.js`. Nécessite un rebuild client complet (nouveaux hash de chunks). Les stats et le compteur sont corrects.

2. **Sync orders Octopia** : L'erreur `response.orders is not iterable` persiste dans `lastSyncError` de `marketplace_octopia_accounts`. C'est un problème séparé (sync commandes, pas discussions).

3. **Sync messages granulaire** : Le sync incrémental met à jour les statuts des discussions existantes mais **n'importe pas les nouveaux messages** pour les discussions déjà importées. Seules les nouvelles discussions reçoivent tous leurs messages lors de l'import initial.
