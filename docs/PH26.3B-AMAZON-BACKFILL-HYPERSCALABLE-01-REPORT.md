## 2. Quotas Retenus + Justification

| Parametre | Valeur DEV | Justification |
|---|---|---|
| `SCHEDULER_MAX_PARALLEL` | 3 | Limite le nombre de tenants actifs simultanement. 3 est suffisant pour DEV avec 2 gros tenants + 1 slot libre. |
| `MAX_ORDERS_CALLS/min/tenant` | 60 | SP-API Orders = ~6 appels/chunk (100 orders/page, ~600 orders/chunk). 60/min = 1 appel/sec avec marge. |
| `MAX_ITEMS_CALLS/min/tenant` | 30 | SP-API OrderItems = 1 appel/order. 30/min = 1 appel/2sec, tres conservateur. |
| `MAX_CALLS_GLOBAL` | 120 | Somme de tous tenants. Empeche l'explosion si 3 tenants font des appels simultanes. |
| `COOLDOWN_BASE_MINUTES` | 5 | Premier cooldown = 5 min. Exponentiel: 2eme = 10 min, 3eme = 15 min... max 60 min. |
| `STALE_ACTIVE_MINUTES` | 15 | Reset automatique si un worker crash sans liberer son slot. |
| `SCHEDULER_TICK_MS` | 5000 | Le scheduler verifie l'etat toutes les 5 secondes. |

## 3. Preuves Fairness Multi-Tenant

### Scenario test
- **ecomlg-001** (GROS) : 10 393 orders Amazon, backfill a 45%
- **ludovic-ml5koskd** (PETIT) : 10 532 orders Amazon, items pending

### Preuves logs

**Scheduler — assignation round-robin :**
```
[Scheduler] ASSIGN_ORDERS tenant=ecomlg-001
[Scheduler] ASSIGN_ITEMS tenant=ludovic-ml5koskd
[Scheduler] GLOBAL running=2 qOrders=7 qItems=6 cooldown=0 done=0
```

**Scheduler — GLOBAL_THROTTLE (fairness en action) :**
```
[Scheduler] GLOBAL_THROTTLE totalCalls=1758 > max=120
```
Le scheduler a mis ecomlg-001 en cooldown apres avoir consomme trop de quota global, laissant ludovic-ml5koskd continuer normalement.

**Scheduler — COOLDOWN_RESET :**
```
[Scheduler] COOLDOWN_RESET orders=1 items=0
```
Le cooldown s'est reset automatiquement apres la duree configuree.

**Orders worker — processing apres scheduler assignment :**
```
[OrdersWorker] CLAIMED tenant=ecomlg-001 worker=amazon-orders-worker-fcfd9b69-mbgf5 status=PENDING scheduler=true
[OrdersWorker] CHUNK tenant=ecomlg-001 from=2024-11-25 to=2024-12-25
[OrdersWorker] FETCHED tenant=ecomlg-001 orders=1756
[OrdersWorker] PERSISTED tenant=ecomlg-001 orders=1756 in 62340ms
```

### Conclusion fairness
- Le petit tenant n'est JAMAIS bloque par le gros
- Le scheduler impose un plafond global et met les gros consommateurs en cooldown
- L'assignation round-robin (lastSuccessAt ASC) garantit que le tenant le moins recemment servi est prioritaire

## 4. Preuves Rate-Limit Handling

**Mecanisme implementé :**
1. Worker detecte un 429 Amazon -> `fetchOrdersPage()` retourne `{ rateLimited: true }`
2. Worker appelle `onOrdersRateLimit(tenantId)` -> scheduler met le tenant en cooldown
3. Scheduler selectionne le prochain tenant eligible (round-robin)
4. Apres expiration du cooldown, le tenant est re-assigne

**Preuve : ecomlg-001 en ERROR apres rate limit**
```
[OrdersWorker] RATE_LIMITED tenant=ecomlg-001 backoff=8000ms attempt=1/5
[OrdersWorker] ERROR tenant=ecomlg-001: Max retries exceeded for SP-API
```
Le worker a signale le cooldown au scheduler, qui a suspendu ecomlg-001 et continue avec ludovic-ml5koskd.

**Backoff exponentiel :**
- 1er cooldown : 5 minutes
- 2eme cooldown : 10 minutes
- 3eme cooldown : 15 minutes
- Maximum : 60 minutes

## 9. Plan de Rollback

### Rollback DEV (vers PH26.3)

```bash
NS="keybuzz-backend-dev"
OLD="ghcr.io/keybuzzio/keybuzz-backend:v1.0.40-ph263-safe-dev"

# Supprimer le scheduler
kubectl delete deployment backfill-scheduler -n ${NS}

# Rollback les workers et backend

kubectl set image deployment/amazon-orders-worker worker="${OLD}" -n ${NS}
kubectl set image deployment/amazon-items-worker worker="${OLD}" -n ${NS}
```

## 11. Limitations Connues

1. **Vault DOWN** : Les tenants dont les credentials Amazon sont dans Vault et non caches ne peuvent pas etre traites par le items worker. Concerne potentiellement `ludovic-ml5koskd`.
2. **ecomlg-001 rate limited** : Les 2 orders restantes sans items retournent 400 de Amazon (probablement des commandes supprimees/invalides). Non lié au scheduler.
3. **Throughput items** : Le items worker est volontairement conservateur (30 appels/min/tenant) pour eviter les 429.
