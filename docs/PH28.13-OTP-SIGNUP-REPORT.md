# PH28.13 - Email OTP Signup Report

**Date**: 2026-02-02  
**Objectif**: Permettre la création de compte via OTP email (sans SSO Google/Microsoft)  
**Statut**: ✅ TERMINÉ - DEV + PROD

---

## 1. Résumé

Cette tâche a permis d'implémenter un flow de signup complet par OTP email :
- Un utilisateur peut créer un compte uniquement avec son adresse email
- L'OTP est envoyé par email (ou affiché en mode DEV)
- Après validation de l'OTP, l'utilisateur et son tenant sont créés
- Redirection automatique vers l'onboarding puis le dashboard

---

## 2. Modifications Backend

### Endpoint existant utilisé: `POST /tenant-context/create-signup`

```
POST /tenant-context/create-signup
Headers: X-User-Email: <email>
Body: {
  name: string,           // Nom du tenant
  slug: string,           // Slug unique
  firstName: string,
  lastName: string,
  country: string,
  supportEmail: string,
  cguAccepted: boolean
}
Response: {
  success: true,
  user: { id, email, name },
  tenant: { id, name, slug },
  membership: { role }
}
```

**Comportement**:
- Crée l'utilisateur si n'existe pas
- Crée le tenant avec les paramètres fournis
- Crée la membership (role: owner)
- Retourne les informations créées

## 3. Modifications Frontend

### Fichier: `keybuzz-client/app/signup/page.tsx`

**Flow implémenté (5 étapes)**:

1. **email**: Saisie de l'email
2. **code**: Validation de l'OTP envoyé par email
3. **profile**: Saisie du profil (prénom, nom, société, pays, CGU)
4. **success**: Création du compte et du tenant
5. Redirection vers `/onboarding`

**Logique clé - `handleVerifyCode`**:
```typescript
// Ne consomme PAS l'OTP ici (sera fait par signIn)
// Passe simplement à l'étape profil
setStep('profile');
```

**Logique clé - `handleCreateTenant`**:
```typescript
// 1. Établir la session NextAuth (consomme l'OTP)
const signInResult = await signIn('email-otp', {
  redirect: false,
  email,
  code,
});

if (!signInResult?.ok) {
  // OTP invalide ou expiré
  setStep('email');
  return;
}

// 2. Créer le tenant via l'API
const response = await fetch('/api/auth/create-signup', {
  method: 'POST',
  body: JSON.stringify({
    name: companyName,
    slug,
    firstName,
    lastName,
    country,
    supportEmail: supportEmail || email,
    cguAccepted: true,
  }),
});

// 3. Redirection
router.push('/onboarding');
```

### Points techniques importants

1. **OTP non consommé prématurément**: L'étape "code" ne consomme plus l'OTP via `/api/auth/magic/verify`. L'OTP est conservé jusqu'à l'appel `signIn('email-otp')`.

2. **Session NextAuth créée AVANT création tenant**: Assure que l'utilisateur est authentifié avant de créer son espace.

3. **UI pour emails non whitelistés (PROD)**: Message "connexion par code pas encore disponible" avec options Google/Microsoft.

## 8. Points d'attention

### Comportement actuel du wizard onboarding

Après signup OTP, l'utilisateur est redirigé vers `/onboarding` qui affiche un wizard de configuration. Ce wizard tente de créer un second tenant, ce qui échoue car le tenant existe déjà.

**Solution**: Naviguer directement vers `/` (dashboard) fonctionne car le tenant a été créé lors du signup.

**Amélioration future**: Le wizard `/onboarding` devrait détecter si un tenant existe déjà et rediriger vers le dashboard.

### Mode DEV temporaire

Pour les tests PROD, `OTP_DEV_MODE=true` a été temporairement activé pour voir les codes OTP dans les logs. Cette variable devrait être retirée après les tests :

```bash
kubectl set env deployment/keybuzz-client OTP_DEV_MODE- --namespace=prod
```
