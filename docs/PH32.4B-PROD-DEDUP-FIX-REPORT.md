# PH32.4B — Fix PROD Messages Not Arriving (Dedup Collision)

## Date
2026-02-12

## Probleme
Les messages Amazon inbound arrivaient en DEV mais **pas en PROD** malgre un webhook fonctionnel (HTTP 200).

## Cause Racine
Les backends DEV et PROD partagent la **meme base de donnees `keybuzz_backend`** pour la deduplication des emails (table `ExternalMessage`).

**Architecture avant fix :**
```
mail-core-01 → postfix_webhook.sh
  ├── POST backend-dev.keybuzz.io (DEV backend) → messageId: "abc123"
  │     └── Check ExternalMessage in keybuzz_backend → NOT FOUND → INSERT + process → write to keybuzz DB ✓
  └── POST backend.keybuzz.io (PROD backend)    → messageId: "abc123"  (SAME!)
        └── Check ExternalMessage in keybuzz_backend → FOUND (by DEV!) → "Already processed" → SKIP ✗
```

### Details techniques
- `keybuzz-backend-dev` : `DATABASE_URL=...keybuzz_backend` / `PRODUCT_DATABASE_URL=...keybuzz`
- `keybuzz-backend-prod` : `DATABASE_URL=...keybuzz_backend` / `PRODUCT_DATABASE_URL=...keybuzz_prod`
- Les deux backends partagent `keybuzz_backend` pour la table de dedup `ExternalMessage` (7723 entries)
- DEV traite toujours en premier → insere le messageId → PROD le trouve deja et skip

### Chronologie
1. Le script webhook PH32.3 PROD-B a ete deploye avec envoi simultane DEV + PROD
2. Les permissions du fichier `/opt/keybuzz/secrets/webhook_key_prod` etaient en `600` (inaccessible)
3. Fix permissions `644` → webhook envoie a PROD avec HTTP 200
4. Mais le backend PROD dit "Already processed" → messages non inseres en `keybuzz_prod`

## Fix Applique

### 1. Webhook (mail-core-01) — messageId isole par environnement
**Fichier :** `/usr/local/bin/postfix_webhook.sh`
**Backup :** `/usr/local/bin/postfix_webhook.sh.bak.ph324b`

Le webhook genere maintenant **deux payloads JSON distincts** avec des messageId uniques :
- DEV payload : `messageId = "original-id-dev"`
- PROD payload : `messageId = "original-id-prod"`

Ainsi les deux backends inserent dans `ExternalMessage` avec des cles differentes et traitent chacun leur email independamment.

### 2. Synchronisation des messages manquants
- **5 conversations** copiees de `keybuzz` (DEV) vers `keybuzz_prod` (PROD)
- **26 messages** copies
- Metadonnees des conversations mises a jour (last_message_preview, updated_at, etc.)

Creer une base de donnees `keybuzz_backend_prod` separee pour le backend PROD afin d'isoler completement les environnements. Cela necessite un acces superuser PostgreSQL (`postgres` role) qui n'est actuellement pas disponible (Vault hors-ligne).
