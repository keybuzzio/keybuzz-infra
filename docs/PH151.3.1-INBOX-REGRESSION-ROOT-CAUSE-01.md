## Verdict

**INBOX REGRESSION ROOT CAUSE IDENTIFIED — SCOPE CONTAMINATION**

---

## Cause racine

Le build PH151.3 (`v3.5.64`) a inclus **243 lignes de changements** dans `InboxTripane.tsx` alors que le scope PH151.3 (priority UX consolidation) ne nécessitait que ~40 lignes de modifications.

**162 lignes supprimées, 81 insérées — dont ~200 lignes hors scope PH151.3.**

La cause : le fichier local `InboxTripane.tsx` contenait des modifications PH151 non commitées (développées localement mais jamais versionnées). Quand PH151.3 a été codé par-dessus, TOUTES les modifications locales ont été incluses dans le SCP vers le bastion et donc dans le build Docker.

## Ligne de code fautive

Le point de rupture le plus dangereux est la suppression du rendu inline des messages (lignes ~1637-1750 de la version commitée) et son remplacement par :

```typescript
<MessageBubble
  key={msg.id}
  msg={{
    ...msg,
    channel: selectedConversation.channel,
    conversationType: selectedConversation.conversationType,
    tenantId: currentTenantId || '',
    collapsed: hideAmazonAuto,
  }}
/>
```

Ce spread operator fusionne l'objet `msg` avec des propriétés additionnelles. Si un message a des propriétés inattendues (ex: `direction` null, `visibility` undefined), le `MessageBubble` peut crasher silencieusement, ce qui cause un écran blanc React (erreur non rattrapée sans ErrorBoundary).

## Scénario de reproduction

1. Déployer `v3.5.64-ph151.3-priority-clean-dev`
2. Naviguer vers `/inbox`
3. La liste de conversations charge (les changements PH151.3 sont cosmétiques ici)
4. Cliquer sur une conversation
5. Le panneau détail tente de rendre :
  - `<ConversationSummaryBar>` avec `classifyInboundMessage()` sur chaque message
  - `<AICaseSummary>` avec `extractCaseSummary()` sur tous les messages
  - `<MessageBubble>` pour chaque message (au lieu du rendu inline)
6. Si un message a un edge case non géré → crash React → écran blanc

### Conditions déclenchantes probables

- Conversation avec messages `direction` ou `visibility` atypiques
- Message avec `content` null (non géré par `classifyInboundMessage` malgré le guard `|| ''`)
- Conversation sans messages (`messages[0]?.timestamp` → `createdAt` undefined dans `ConversationSummaryBar`)
- Messages avec `messageSource` inconnu du classifier

## Pourquoi le CE a validé à tort

### Ce qui a été vérifié

- L'Inbox chargeait 378 conversations ✓
- Les badges PH151.3 étaient corrects ✓
- Les filtres fonctionnaient ✓
- Un screenshot montrait les conversations ✓

### Ce qui n'a PAS été vérifié

1. **Le diff complet n'a jamais été audité** — le CE n'a pas comparé le diff réel avec le scope PH151.3
2. **Aucune vérification console** — les erreurs JavaScript n'ont pas été inspectées
3. **Pas de test multi-conversations** — seule la conversation par défaut a été vue
4. **Pas de vérification des fichiers SCP'd** — les 3 fichiers SCP'd pour résoudre les erreurs de build (ConversationActionBar, AgentWorkbenchBar, ConversationSummaryBar) auraient dû alerter : si le build échoue sur des composants manquants, c'est que le scope dépasse PH151.3
5. **Le nombre de fichiers SCP'd n'a pas été questionné** — SCP de 3 fichiers pour résoudre des erreurs de build est un signal d'alerte majeur

### Signal d'alerte ignoré

Les 3 erreurs de build successives étaient un **signal clair** de scope contamination :

- Erreur 1 : `ConversationActionBar` manquant → SCP'd
- Erreur 2 : `AgentFilterKey` type mismatch → SCP'd `AgentWorkbenchBar`
- Erreur 3 : `ConversationSummaryBarProps.status` manquant → SCP'd `ConversationSummaryBar`

**Ces fichiers n'avaient RIEN à voir avec PH151.3 (priority UX).** Leur absence/incompatibilité prouvait que le `InboxTripane.tsx` local contenait des dépendances hors scope.

## Recommandations (à ne PAS exécuter dans cette phase)

1. **Toujours vérifier `git diff --stat` AVANT de SCP** — si plus de fichiers sont modifiés que le scope, STOP
2. **Si un build échoue sur un composant hors scope, STOP et auditer** — ne jamais SCP des fichiers supplémentaires sans comprendre pourquoi
3. **Les fichiers PH151 (MessageBubble, AICaseSummary, etc.) doivent être traités dans une phase dédiée**, avec tests complets et commit Git propre
4. **Ajouter un ErrorBoundary autour du panneau détail conversation** pour éviter les écrans blancs
