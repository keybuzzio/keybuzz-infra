## 4. Cause racine finale

**DOUBLE BUG dans `InboxTripane.tsx`** :

### Bug 1 : Pas de rechargement après envoi
`handleSendReply()` (le handler pour le textarea/reply box classique) **ne rechargeait PAS** la conversation après l'envoi. Il mettait uniquement à jour le message optimiste (ID temporaire → ID réel) et affichait un toast "Réponse envoyée".

**Comparaison** : `onDirectSend()` (le handler Autopilot) appelait `loadConversationDetail(selectedId)` après l'envoi — c'est pourquoi l'escalade fonctionnait en Autopilot mais pas en envoi manuel.

### Bug 2 : Cache local bloquant le refresh
`loadConversationDetail()` contenait un **cache local** qui retournait les données déjà en mémoire si la conversation existait :
```javascript
const localConv = conversations.find(c => c.id === id);
if (localConv && localConv.messages.length > 0) {
  setSelectedConversation(localConv); // ← CACHE, pas l'API !
  return; // ← Court-circuite l'appel API
}
```
Même si on ajoutait `loadConversationDetail()` après le reply, il retournait le cache avec l'ancien `escalation_status: "none"`.

## 5. Correction appliquée

### Fichier modifié
`keybuzz-client/app/inbox/InboxTripane.tsx`

### Fix 1 : Paramètre `forceRefresh` sur `loadConversationDetail`
```typescript
const loadConversationDetail = useCallback(async (id: string, forceRefresh?: boolean) => {
  // ...
  const localConv = conversations.find(c => c.id === id);
  if (!forceRefresh && localConv && localConv.messages.length > 0) {
    setSelectedConversation(localConv);
    setIsLoadingDetail(false);
    return;
  }
  // Si forceRefresh=true, on fetch depuis l'API
  // ...
```

### Fix 2 : Appel loadConversationDetail(id, true) après reply
```typescript
// Dans handleSendReply, après sendReply réussi :
if (selectedId && visibility !== 'internal_note') {
  setTimeout(() => loadConversationDetail(selectedId, true), 800);
}
```

### Fix 3 : onDirectSend utilise aussi forceRefresh
```typescript
setTimeout(() => loadConversationDetail(selectedId, true), 800);
```
