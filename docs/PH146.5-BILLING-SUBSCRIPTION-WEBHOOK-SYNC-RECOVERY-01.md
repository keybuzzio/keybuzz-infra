---

## 1. Reproduction

**Constat** : 10/10 subscriptions dans `billing_subscriptions` avaient `current_period_end = NULL`.

```
switaa-sasu-mnc1x4eq    | sub=sub_1TGNQm... | status=active   | end=NULL
compta-ecomlg-gmail--   | sub=sub_1TIlQV... | status=trialing | end=NULL
switaa-mn9ioy5j         | sub=sub_1TFjZL... | status=active   | end=NULL
...
(10/10 NULL)
```

La colonne `current_period_start` n'existait pas dans le schema.

---

## 2. Audit Webhook

Le webhook handler est correctement configure :
- `customer.subscription.created` -> `handleSubscriptionChange()` OK
- `customer.subscription.updated` -> `handleSubscriptionChange()` OK
- `customer.subscription.deleted` -> `handleSubscriptionDeleted()` OK
- `checkout.session.completed` -> `handleCheckoutCompleted()` OK (email seulement)

Evenements recus : 105 `subscription.updated`, 48 `subscription.created`, 47 `checkout.session.completed`.

---

## 3. Cause Racine

**L'API Stripe a supprime `current_period_end` et `current_period_start` du top-level de l'objet Subscription dans les versions recentes.**

Preuve — payload webhook stocke en DB :

```
=== Top-level keys in event.data.object ===
  billing_cycle_anchor: 1776022844
  status: "active"
  trial_end: 1776022844
  ...
  (PAS de current_period_end)
  (PAS de current_period_start)
```

**Ces champs sont maintenant dans `items.data[0]` :**

```
Item[0]:
  current_period_end: 1778614844    (2026-05-12T19:40:44.000Z)
  current_period_start: 1776022844  (2026-04-12T19:40:44.000Z)
```

Le code ancien faisait `subscription.current_period_end` qui retournait `undefined`.
`to_timestamp(undefined)` en PostgreSQL produit `NULL`.

---

## 4. Fix Applique

**Fichier modifie** : `src/modules/billing/routes.ts`

### 4.1 Extraction des periodes (handleSubscriptionChange)

```typescript
// PH146.5: Extract period from items.data
const firstItem = subscription.items?.data?.[0];
const periodEnd = (subscription as any).current_period_end
  || (firstItem as any)?.current_period_end
  || subscription.trial_end
  || null;
const periodStart = (subscription as any).current_period_start
  || (firstItem as any)?.current_period_start
  || subscription.trial_start
  || null;
```

Fallback chain : top-level -> items.data[0] -> trial dates -> null

### 4.2 Query INSERT/UPDATE

```sql
INSERT INTO billing_subscriptions 
  (..., current_period_end, current_period_start)
  VALUES (..., to_timestamp($9), to_timestamp($10))
ON CONFLICT (tenant_id) DO UPDATE SET
  ...
  current_period_end = to_timestamp($9),
  current_period_start = to_timestamp($10),
  updated_at = NOW()
```

### 4.3 Query SELECT (billing/current endpoint)

Ajout de `current_period_start` dans les 2 queries SELECT et le response mapping.

### 4.4 Schema DB

```sql
ALTER TABLE billing_subscriptions ADD COLUMN IF NOT EXISTS current_period_start TIMESTAMPTZ;
```

### 4.5 Type BillingData

```typescript
currentPeriodStart?: string | null;
```

---

## 5. Script Backfill

Script idempotent executee une fois post-deploy :
- Lit toutes les `billing_subscriptions` de la DB
- Pour chaque subscription, appelle `stripe.subscriptions.retrieve(subId, { expand: ['items.data'] })`
- Extrait `current_period_end` et `current_period_start` depuis items.data[0]
- UPDATE en DB avec `to_timestamp()`

Resultat :

```
Updated: 10
Skipped: 0
Errors: 0
Total: 10
```
