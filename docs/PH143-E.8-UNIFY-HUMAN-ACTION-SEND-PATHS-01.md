## 1. Résumé exécutif

Cette phase corrige le **dernier trou** dans le workflow Autopilot/IA : un message contenant une promesse d'action humaine pouvait être envoyé **sans déclencher d'escalade** selon le chemin UI utilisé. Le fix ajoute un **filet de sécurité côté serveur** dans le handler `POST /conversations/:id/reply` qui détecte les fausses promesses et déclenche l'escalade automatiquement, quel que soit le chemin d'envoi.

---

## 2. Cartographie des chemins UI d'envoi

| Chemin | Composants | Route API | Consume | Escalade (avant fix) |
|--------|-----------|-----------|---------|---------------------|
| **A — Brouillon Autopilot** | `AISuggestionSlideOver` → `onDirectSend` + `consumeDraft('applied')` | `POST /reply` + `POST /autopilot/draft/consume` | OUI | OUI (via consume, E.6) |
| **B — Suggestion IA** | `AISuggestionSlideOver` → `onInsertResponse` → zone réponse → `handleSendReply` | `POST /reply` uniquement | NON | **NON** |
| **C — Réponse directe** | Zone réponse → `handleSendReply` | `POST /reply` uniquement | NON | **NON** |

**Problème** : les chemins B et C contournent le pipeline `consume` et ne vérifient pas les promesses d'action humaine.

---

## 3. Cause racine finale

Le pipeline `consume` (corrigé en E.6) ne couvre que le chemin A (brouillon Autopilot validé). Les chemins B (suggestion IA insérée puis envoyée manuellement) et C (réponse tapée directement) passent uniquement par `POST /conversations/:id/reply` sans aucune vérification de contenu.

**Solution** : ajouter la détection de fausses promesses **directement dans le handler reply** — point de passage obligatoire de TOUS les chemins.

---

## 4. Correction appliquée

### Fichier modifié
`keybuzz-api/src/modules/messages/routes.ts` — handler `POST /conversations/:id/reply`

### Logique ajoutée (lignes 533-583)
Après l'envoi du message (insertion DB + outbound delivery queued) et avant le retour :

1. **Détection** : 13 patterns regex (mêmes que `detectFalsePromises` dans engine.ts) scannent le contenu de la réponse
2. **Vérification** : si des promesses sont détectées ET que la conversation n'est pas déjà escaladée (`escalation_status != 'escalated'`)
3. **Escalade** : UPDATE `conversations` avec `escalation_status = 'escalated'`, `escalation_target = 'client'`, `escalation_reason` décrivant les promesses détectées
4. **Event** : INSERT dans `message_events` (type `auto_escalate_false_promise`)
5. **Réponse** : champ `escalatedByReply: true/false` ajouté à la réponse API

### Caractéristiques
- **Non-bloquant** : l'escalade est en `try/catch`, une erreur n'empêche pas l'envoi
- **Idempotent** : si la conversation est déjà escaladée, pas de doublon
- **Couvre tous les chemins** : A (Autopilot — double protection), B (Suggestion IA), C (Direct)
- **Ne touche que les messages `outbound`** : les notes internes ne déclenchent pas

## 6. Comportement des anciens drafts

Les drafts générés avant PH143-E.7 qui étaient mal classés comme `DRAFT_GENERATED` (alors qu'ils contenaient des promesses d'action) sont maintenant **couverts** par le filet de sécurité serveur :

- Si un tel draft est validé et envoyé via `onDirectSend` → le consume ne déclenche pas d'escalade (car `DRAFT_GENERATED`), **MAIS** le handler reply détecte les fausses promesses et escalade automatiquement
- Si le texte est inséré dans la zone de réponse et envoyé manuellement → même protection

**Résultat** : aucun message avec promesse d'action humaine ne peut désormais échapper à l'escalade, quelle que soit la version du draft ou le chemin UI.
