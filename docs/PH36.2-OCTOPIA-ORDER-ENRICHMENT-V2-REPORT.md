## 1. Endpoints Octopia utilisés

| Endpoint | Données extraites |
|----------|------------------|
| `GET /orders/{orderId}` | Tout : produits, prix, statut, client, téléphone, tracking, commission, frais, timeline |

Pas d'endpoint `financial`, `events` ou `history` séparé dans l'API Octopia v2. Toutes les données proviennent de la réponse principale de l'ordre.

### Champs sources dans la réponse Octopia

| Donnée | Chemin JSON |
|--------|-------------|
| Téléphone client | `lines[0].shippingAddress.phone` |
| Email client (proxy) | `lines[0].shippingAddress.email` |
| Statut commande | `status` (ordre) + `lines[].status` (ligne, plus précis) |
| Prix vente | `totalPrice.sellingPrice` |
| Commission | `lines[].offerPrice.commission.amountWithVat` |
| Frais de service | `serviceFees[].amount` |
| Nom client | `billingAddress.firstName` + `billingAddress.lastName` |
| Adresse | `billingAddress.addressLine1`, `postalCode`, `city`, `countryCode` |
| Tracking | `lines[].parcels[].carrierName`, `parcelNumber`, `trackingUrl` |
| Date achat | `purchasedAt` |
| Date expédition | `shippedAtMax` |
| Date mise à jour | `updatedAt` |

## 2. Modèle générique (MP-friendly)

Structure ajoutée à la réponse API de chaque commande :

```json
{
  "marketplaceStatus": {
    "label": "Livrée",
    "step": 5
  },
  "customerPhone": "0680993317",
  "timeline": [
    { "at": "2026-01-29T20:54:26.943Z", "label": "Commande passée" },
    { "at": "2026-02-04T21:55:19.16Z", "label": "Expédiée (DB Schenker)" },
    { "at": "2026-02-17T17:36:56.21Z", "label": "Livrée" }
  ],
  "financialSummary": {
    "currency": "EUR",
    "sale": 890.07,
    "commission": -52.02,
    "serviceFees": -22.89,
    "refunds": 0,
    "total": 815.16
  }
}
```

Pour Amazon : `marketplaceStatus`, `timeline`, `financialSummary` et `customerPhone` retournent `null` / `[]` — pas de régression.

### Statut marketplace — mapping

| Code Octopia | Label FR | Step |
|-------------|----------|------|
| Created | Créée | 1 |
| AcceptedBySeller | Acceptée | 2 |
| Shipped | Expédiée | 3 |
| InTransit | En transit | 4 |
| Delivered | Livrée | 5 |
| Cancelled | Annulée | -1 |
| Refunded | Remboursée | -2 |

Logique : si **toutes** les lignes sont `Delivered`, le statut effectif est `Delivered` (step 5), même si l'ordre reste à `Shipped`.

## 3. Fichiers modifiés

### API — `keybuzz-api/src/modules/orders/routes.ts`

- **`mapOctopiaOrderToDb`** : enrichi avec extraction phone, email, marketplaceStatus, timeline, financialSummary depuis la réponse Octopia
- **`orderRowToApiResponse`** : appelle `extractEnrichedFields(row)` qui reconstruit les champs enrichis depuis `raw_data` (JSONB stocké à l'import)
- **`extractEnrichedFields`** : fonction générique qui vérifie `channel === 'octopia'` puis extrait les données enrichies depuis `raw_data.lines`, `raw_data.totalPrice`, etc.

### Client — `keybuzz-client/src/features/inbox/components/OrderSidePanel.tsx`

- **Interface `OrderSummary`** : ajout de `customerPhone`, `marketplaceStatus`, `timeline`, `financialSummary`
- **Imports** : ajout `Phone`, `DollarSign`, `History` depuis lucide-react
- **Sections UI ajoutées** :
  - **Statut marketplace** : badge coloré + barre de progression (5 steps)
  - **Client > Téléphone** : affiché sous le nom avec lien `tel:`
  - **Historique** : timeline verticale avec points et dates
  - **Bilan financier** : tableau avec Vente, Commission, Frais, Total net
