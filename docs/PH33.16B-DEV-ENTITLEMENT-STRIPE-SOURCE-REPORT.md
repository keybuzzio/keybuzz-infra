# PH33.16B — Trial/grace alignés Stripe (source de vérité billing_subscriptions)

## Objectif
Remplacer le calcul trial basé uniquement sur `tenants.created_at` par une chaîne de priorité alignée Stripe :

1. `billing_subscriptions.trial_end` (Stripe — si la colonne existe)
2. `billing_subscriptions.current_period_end` (Stripe)
3. `tenants.created_at + 14 jours` (fallback)

---

## Ce qui a été fait

### Service `getTenantEntitlement` réécrit

Le service détecte **dynamiquement** si la colonne `trial_end` existe dans `billing_subscriptions` (résultat mis en cache) :

```
billing_subscriptions.trial_end existe ?
├── OUI → SELECT plan, status, current_period_end, trial_end, updated_at
└── NON → SELECT plan, status, current_period_end, updated_at
```

#### Priorité trial end :
| # | Source | Condition |
|---|--------|-----------|
| 1 | `billing_subscriptions.trial_end` | Colonne existe ET valeur non null |
| 2 | `billing_subscriptions.current_period_end` | Valeur non null |
| 3 | `tenants.created_at + 14j` | Fallback si aucune donnée Stripe |

Le champ `trialEndsAtSource` dans la réponse indique quelle source a été utilisée.

### Colonne `trial_end`

La colonne n'a **pas pu être ajoutée** car la table `billing_subscriptions` est propriété du user `postgres` (superuser), et aucun des users disponibles (API dev, migrator, vault admin) n'a les privilèges ALTER TABLE.

**Action requise** : Un DBA doit exécuter :
```sql
ALTER TABLE billing_subscriptions ADD COLUMN IF NOT EXISTS trial_end TIMESTAMPTZ;
```

Quand cette colonne sera ajoutée, le code l'utilisera **automatiquement** (détection dynamique, résultat caché).

### Webhook `handleSubscriptionChange`

Le webhook n'a **pas** été modifié (la colonne `trial_end` n'existe pas encore). Quand la colonne sera ajoutée, le webhook devra être mis à jour pour stocker `subscription.trial_end`.
