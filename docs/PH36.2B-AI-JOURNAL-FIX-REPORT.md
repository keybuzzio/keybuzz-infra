# PH36.2B — Fix Journal IA (list + detail) + Anti-garbage events

**Date** : 2026-02-20  
**Env** : DEV ONLY  
**Status** : VALIDÉ  

---

## Cause racine

### Bug 1 : ENTITÉ affiche du texte random
Le script de backfill `ph367_backfill_ai_journal.sh` a injecté le **texte du message** dans le champ `conversation_id` de la table `ai_action_log` au lieu du vrai ID de conversation. Le frontend appelait `.slice(-8)` sur ce texte, d'où les fragments "enir une", "onjour", "ivraison".

### Bug 2 : "Événement introuvable" sur toutes les pages Détail
La page `[eventId]/page.tsx` faisait `fetch(/api/ai/journal?eventId=${eventId})` **sans passer `tenantId`**. Le backend retournait 400 (tenantId required). De plus, le backend `/journal` n'avait pas de filtre par `eventId`.

### Bug 3 : 100+ events "Evaluated 0 rules" identiques
Le backfill créait 1 event par message traité, sans regroupement ni validation de la qualité des données.

---

## Corrections appliquées

### Frontend — Page liste (`app/ai-journal/page.tsx`)
- **ENTITÉ** : validation regex `^(conv-|[0-9a-f]{8,})` avant affichage
  - Si valide : lien cliquable `/inbox?id=<conv_id>` avec `.slice(-12)` 
  - Si invalide : affiche `[channel]` ou `(auto-eval)` en italique gris
- Import `useTenantId` corrigé → `TenantProvider` (au lieu de `useTenantId.ts`)

### Frontend — Page détail (`app/ai-journal/[eventId]/page.tsx`)
- `useTenantId()` importé depuis `TenantProvider`
- `tenantId` passé dans le fetch API
- Validation `isValidEntityId()` : le bouton "Ouvrir la conversation" n'apparaît que si l'entity ID est un vrai identifiant
- Message d'erreur amélioré : "Événement supprimé ou introuvable" + affichage de l'ID

### Backend — Route `/ai/journal` (`modules/ai/routes.js`)
- Ajout du paramètre `eventId` dans le query destructuring
- Ajout clause `WHERE id = $N` quand `eventId` est fourni

### Data — Nettoyage `ai_action_log`
- **209 entrées garbage supprimées** (conversation_id = texte de message)
- **47 entrées valides conservées** (conversation_id = `conv-oct-*`)
- **20 entrées re-backfillées** avec vrais conversation IDs depuis la table `conversations`
- **Total final : 67 entrées propres**
