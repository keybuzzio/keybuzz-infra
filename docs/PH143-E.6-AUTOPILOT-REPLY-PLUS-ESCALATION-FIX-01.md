## 1. Résumé exécutif

Le bug critique du pipeline Autopilot où **la validation d'un brouillon d'escalade envoyait le message au client mais ne déclenchait PAS l'escalade** est corrigé.

La cause racine était dans l'endpoint `POST /autopilot/draft/consume` : quand un `ESCALATION_DRAFT` était consommé avec l'action `applied`, le code mettait à jour le `blocked_reason` en `DRAFT_APPLIED` mais **ne déclenchait jamais la logique d'escalade réelle** (mise à jour de `conversations.escalation_status`, `escalation_target`, etc.).

---

## 2. Reproduction exacte du bug

### Flux observé AVANT le fix

1. IA Autopilot détecte un cas nécessitant investigation humaine
2. Moteur crée un `ESCALATION_DRAFT` avec `{ escalationTarget, escalationReason }` dans le payload `ai_action_log`
3. Utilisateur voit le brouillon dans le drawer
4. Clic "Valider et envoyer"
5. `onDirectSend(text)` → envoie le message client ✅
6. `consumeDraft('applied')` → met à jour `blocked_reason` en `DRAFT_APPLIED` ✅
7. **AUCUNE escalade déclenchée** ❌
8. Conversation reste en `status = pending/open`, `escalation_status = none` ❌

### Résultat : message envoyé, escalade perdue, conversation non visible comme "à reprendre"

---

## 3. Cause racine

**Fichier** : `keybuzz-api/src/modules/autopilot/routes.ts` — endpoint `POST /draft/consume`

Le endpoint `consume` ne faisait que :
1. Mettre à jour `blocked_reason` de `ESCALATION_DRAFT:X` vers `DRAFT_APPLIED`
2. Logger un événement d'audit dans `ai_action_log`
3. Retourner `{ consumed: true, action: 'applied' }`

Il **ne vérifiait jamais** si le draft original était un `ESCALATION_DRAFT` et **ne déclenchait jamais** l'escalade.

En comparaison, en mode **autonome**, le moteur appelle directement `executeReply()` + `escalateConversation()` (ligne 299-300 de `engine.ts`). Mais en mode **safe** (avec validation utilisateur), l'escalade était perdue au moment du consume.

---

## 4. Correction appliquée

### API — `src/modules/autopilot/routes.ts`

Avant de mettre à jour le `blocked_reason`, le code lit maintenant le draft original :

```typescript
const originalDraft = await pool.query(
  `SELECT blocked_reason, payload FROM ai_action_log
   WHERE id = $1 AND tenant_id = $2 AND conversation_id = $3
     AND (blocked_reason = 'DRAFT_GENERATED' OR blocked_reason LIKE 'ESCALATION_DRAFT%')`,
  [logId, tenantId, conversationId]
);
const wasEscalationDraft = originalDraft.rows.length > 0
  && (originalDraft.rows[0].blocked_reason || '').startsWith('ESCALATION_DRAFT');
```

Si `action === 'applied'` ET `wasEscalationDraft === true` :

```typescript
const escalationTarget = payload?.escalationTarget || 'client';
const escalationReason = payload?.escalationReason || 'Escalade automatique apres validation brouillon';

await pool.query(
  `UPDATE conversations
   SET escalation_status = 'escalated',
       escalation_target = $1,
       escalation_reason = $2,
       escalated_at = now(),
       escalated_by_type = 'ai',
       status = 'escalated',
       updated_at = now()
   WHERE id = $3 AND tenant_id = $4`,
  [escalationTarget, escalationReason, conversationId, tenantId]
);
```

Un événement `autopilot_escalate` est aussi inséré dans `message_events`.

La réponse inclut maintenant `{ consumed: true, action: 'applied', escalated: true }`.

### Client — `app/inbox/InboxTripane.tsx`

Après `sendReply` dans `onDirectSend`, un rechargement de la conversation est déclenché :

```typescript
setTimeout(() => loadConversationDetail(selectedId), 800);
```

Ce délai de 800ms permet au backend de traiter le consume + escalade avant que le client ne recharge les données.
