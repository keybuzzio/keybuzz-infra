## Problème identifié

### Goulot d'étranglement: Fetch des OrderItems

Le fetch des items pour chaque commande est le goulot d'étranglement principal:

- **700 commandes** = 140 appels API items (avec concurrency 5)
- **Chaque appel** = ~200ms minimum + rate limiting potentiel
- **Temps estimé** = 140 * 0.5s = **70+ secondes** juste pour les items d'un chunk

### Comportement observé

```
[BackfillFast] FETCHED tenant=ecomlg-001 orders=700
... (worker s'arrête avant ITEMS_FETCHED car timeout/event loop)
```

---

## Solutions proposées

### Option A: Skip items pour backfill initial (RECOMMANDÉ)

1. Backfill initial = orders seulement (TRÈS RAPIDE)
2. Job séparé = fetch items pour orders sans items
3. Delta sync = orders + items comme avant

**Avantage**: x10-x50 plus rapide pour le backfill

### Option B: Reports API

1. Utiliser Reports API pour bulk download orders
2. Parser le fichier TSV/CSV
3. Bulk upsert

**Avantage**: Pas de rate limiting par commande

### Option C: Worker K8s dédié (DÉJÀ PRÉPARÉ)

1. Déployer `deployment-backfill-worker.yaml`
2. Worker loop continu hors du pod principal
3. Pas de timeout HTTP
