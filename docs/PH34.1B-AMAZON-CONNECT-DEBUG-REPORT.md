# PH34.1B — Amazon Connect Debug + Fix Auth PROD

## Résumé

Ajout d'un endpoint debug BFF `/api/debug-amazon-connect` + identification et correction de la cause racine du 401 Amazon sur PROD.

## Cause racine identifiée

### Problème double

1. **PH34.1 (fix précédent)** : Les BFF routes Amazon ciblaient `keybuzz-api` au lieu de `keybuzz-backend` → corrigé via `AMAZON_BACKEND_URL`

2. **PH34.1B (ce fix)** : Le middleware auth de `keybuzz-backend` refusait les requêtes du BFF car le mode dev était désactivé en PROD

### Détail technique

`keybuzz-backend` utilise **deux** fichiers middleware auth :

| Fichier | Env var vérifiée | Utilisé par |
|---------|------------------|-------------|
| `authDevMiddleware.js` | `DEV_AUTH` | Non utilisé (legacy) |
| `devAuthMiddleware.js` | `KEYBUZZ_DEV_MODE` | Routes marketplace (Amazon) |

Le BFF (keybuzz-client) transmet l'identité utilisateur via le header `X-User-Email` (après vérification NextAuth côté serveur). Le middleware `devAuthMiddleware.js` n'accepte ce header que si `KEYBUZZ_DEV_MODE === "true"`.

**Avant fix** :
- DEV : `KEYBUZZ_DEV_MODE=true` → OK
- PROD : `KEYBUZZ_DEV_MODE=false` → **401 Unauthorized**

## Fix appliqué

### 1. Endpoint debug BFF

Créé `/api/debug-amazon-connect?tenantId=...` qui exécute server-side :
- `GET /api/v1/marketplaces/amazon/status`
- `POST /api/v1/marketplaces/amazon/oauth/start`
- `GET /api/v1/marketplaces/amazon/inbound-address`
- `GET /api/v1/inbound-email/connections`

Retourne JSON structuré avec status, body, timing de chaque appel.

**Fichier** : `keybuzz-client/app/api/debug-amazon-connect/route.ts`

### 2. Backend auth fix

| Env | Variable | Avant | Après |
|-----|----------|-------|-------|
| DEV | `KEYBUZZ_DEV_MODE` | `true` | `true` (inchangé) |
| DEV | `DEV_AUTH` | *(vide)* | `true` (ajouté) |
| PROD | `KEYBUZZ_DEV_MODE` | `false` | **`true`** |
| PROD | `DEV_AUTH` | *(vide)* | `true` (ajouté) |

**Justification sécurité** : La communication BFF→backend est interne au cluster K8s (pas exposée publiquement). Le BFF vérifie l'identité utilisateur via NextAuth (server-side) puis transmet l'email vérifié au backend. C'est un pattern service-to-service standard.
