# PH145.8 — Agent First Login Session Recovery

> Date : 12 avril 2026
> Environnement : **DEV uniquement**
> Verdict : **FIX VALIDÉ — 3 corrections appliquées**

---

## Contexte

Le premier login d'un agent invité présentait deux problèmes :

1. **Session chaotique** : l'agent voyait le menu admin complet et des pages vides, nécessitant une déconnexion/reconnexion
2. **UX invitation dégradée** : l'agent devait saisir manuellement son email et cliquer "Envoyer le code" alors que le token d'invitation contenait déjà cette information

## Cause racine

### Problème 1 : Session chaotique (menu admin au premier login)

**Fichier** : `app/invite/continue/page.tsx`

Après acceptation de l'invitation, la page utilisait `router.push()` (navigation SPA) au lieu de `window.location.href` (hard reload). Conséquence : le `TenantProvider` conservait un état stale (récupéré avant l'acceptation de l'invitation), le cookie `currentTenantRole` n'était pas lu par le middleware, et le menu affichait toutes les entrées admin.

### Problème 2 : Email non pré-rempli

**Fichiers** : `app/login/page.tsx`, `app/api/space-invites/resolve/route.ts`, `src/modules/auth/space-invites-routes.ts`, `src/lib/routeAccessGuard.ts`

Le lien d'invitation passait un `invite_token` dans l'URL, mais :

- Aucun endpoint API ne permettait de résoudre ce token sans authentification
- Aucune route BFF `/api/space-invites/resolve` n'existait côté client
- Le middleware Next.js bloquait `/api/space-invites/`* car absent de `API_PUBLIC_PREFIXES`

## Corrections appliquées

### Fix 1 : Hard reload après acceptation invitation (v3.5.251)

```diff
// app/invite/continue/page.tsx — lignes 140 et 170
- router.push('/inbox');
+ window.location.href = '/inbox';

- router.push(destination);
+ window.location.href = destination;
```

Identique au pattern utilisé par `app/select-tenant/page.tsx` qui fonctionnait correctement.

### Fix 2 : Auto-résolution email + envoi OTP (v3.5.252)

```diff
// app/login/page.tsx — nouveau useEffect
+ useEffect(() => {
+   if (!inviteToken || inviteResolvedRef.current || isAuthenticated) return;
+   inviteResolvedRef.current = true;
+   (async () => {
+     const res = await fetch(`/api/space-invites/resolve?token=...`);
+     if (res.ok) {
+       const data = await res.json();
+       if (data.email) {
+         setEmail(data.email);
+         // Auto-send OTP
+         const otpRes = await fetch(`/api/auth/magic/start`, { ... });
+         if (otpRes.ok) setStep('code');
+       }
+     }
+   })();
+ }, [inviteToken, isAuthenticated]);
```

### Fix 3 : Endpoint resolve + middleware public (v3.5.253 + API v3.5.48)

**Backend API** (`space-invites-routes.ts`) :

```typescript
app.get('/resolve', async (request, reply) => {
  const token = (request.query as any).token;
  const tokenHash = hashToken(token);
  const result = await pool.query(
    `SELECT si.email, si.role, t.name as tenant_name
     FROM space_invites si JOIN tenants t ON si.tenant_id = t.id
     WHERE si.token_hash = $1 AND si.accepted_at IS NULL AND si.expires_at > NOW()`,
    [tokenHash]
  );
  // Returns { email, tenantName, role } — no auth required
});
```

**Client BFF** (`app/api/space-invites/resolve/route.ts`) :

- Proxy vers l'API backend, sans authentification

**Middleware** (`routeAccessGuard.ts`) :

```diff
  API_PUBLIC_PREFIXES = [
    ...
+   '/api/invite',
+   '/api/space-invites',
    '/debug/version',
  ];
```

## Sécurité

L'endpoint `GET /space-invites/resolve` est sans authentification par design :

- Utilisé avant le login pour pré-remplir l'email
- Ne retourne que l'email, le nom du tenant et le rôle — aucune donnée sensible
- Le token est un hash SHA-256 de 64 caractères, non devinable
- Seules les invitations non-acceptées et non-expirées sont retournées
- Equivalent en sensibilité à l'affichage de l'email sur la page d'invitation elle-même
