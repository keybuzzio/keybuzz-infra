## 2. Causes racines (3 problemes independants)

### CR1 : CronJobs sans X-Internal-Token
- **Symptome** : `{"error":"Unauthorized","message":"User not found in database"}`
- **Cause** : Les CronJobs `amazon-orders-sync` et `amazon-orders-backfill` envoyaient `X-User-Email: cron@system.keybuzz.io` mais cet utilisateur n'existe pas dans la DB Prisma du backend. Le `devAuthMiddleware` rejetait la requete en 401.
- **Fix** : Ajout du header `X-Internal-Token` (lu depuis le secret K8s `keybuzz-internal-proxy`) dans les deux CronJobs. Le middleware proxy bypass le user DB lookup.

### CR2 : SP-API v2026-01-01 parameter case mismatch
- **Symptome** : `SP-API error 400: InvalidInput — One and only one of createdAfter or lastUpdatedAfter must be provided`
- **Cause** : Le code API (`orders/routes.ts`) utilisait l'URL v2026-01-01 (`/orders/2026-01-01/orders`) mais avec les noms de parametres PascalCase de l'ancienne API v0 :
  - `MarketplaceIds` → attendu `marketplaceIds`
  - `CreatedAfter` → attendu `createdAfter`
  - `NextToken` → attendu `paginationToken`
  - Response: `data.Orders` → attendu `data.orders`
  - Response: `data.NextToken` → attendu `data.pagination.nextToken`
- **Fix** : 5 corrections de noms de parametres dans `fetchAmazonOrdersPage()`.

### CR3 : Backfill CronJob URL inexistante
- **Symptome** : `404 — Route POST:/api/v1/orders/backfill/run/global not found`
- **Cause** : L'endpoint `/run/global` n'existe pas dans les routes backfill. L'endpoint correct est `/worker/run`.
- **Fix** : URL corrigee dans le manifest CronJob.

  - `refresh_token` : present (Atzr|IwEBINBB5...)
  - `seller_id` : A12BCIS2R7HD4D

L'ancien probleme Vault concernait le service `vault.service` systemd sur la VM, pas le pod K8s.

## 4. Fix applique

### 4.1 CronJob `amazon-orders-sync`
- Ajout env `INTERNAL_TOKEN` depuis secret `keybuzz-internal-proxy`
- Ajout header `X-Internal-Token: $INTERNAL_TOKEN`
- Image fixee : `curlimages/curl:8.12.1` (plus de `:latest`)

### 4.2 CronJob `amazon-orders-backfill`
- Meme fix auth (INTERNAL_TOKEN + header)
- Ajout `-d '{}'` pour eviter `FST_ERR_CTP_EMPTY_JSON_BODY`
- URL corrigee : `/backfill/worker/run` (au lieu de `/backfill/run/global`)
- Image fixee : `curlimages/curl:8.12.1`

### 4.3 API `orders/routes.ts`
- `MarketplaceIds` → `marketplaceIds`
- `CreatedAfter` → `createdAfter`
- `NextToken` → `paginationToken`
- Response: `data?.orders` (avec fallback `data?.Orders`)
- Response: `data?.pagination?.nextToken` (avec fallback `data?.NextToken`)

## 8. Issues connues restantes

### Backend delta sync (SP-API date params)
Le service `amazonOrdersSync.service.js` du backend a le meme probleme de parametres SP-API v2026 vs v0. Le CronJob global sync echoue pour les tenants avec tokens. Ce fix necessite un rebuild du backend (hors scope PH145.3).

### BFF Amazon status route
Le BFF `app/api/amazon/status/route.ts` appelle `BACKEND_URL/api/v1/marketplaces/amazon/status` mais cette route n'existe pas dans l'API Fastify. La route existe dans le backend (keybuzz-backend) mais le BFF pointe vers l'API. Impact : le statut Amazon dans l'UI channels peut etre incorrectement affiche.

### Backfill scheduler throttling
Le scheduler long-running montre `GLOBAL_THROTTLE totalCalls=1776 > max=120`. Il est rate-limite par sa propre configuration, ce qui ralentit l'import. Non bloquant mais sous-optimal.
