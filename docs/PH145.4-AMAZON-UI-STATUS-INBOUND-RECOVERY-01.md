## 2. Verite historique retrouvee

### Chaine complete OAuth → Channels → Orders

```
1. User ajoute Amazon depuis le catalogue
   → addChannel() met status = 'pending' (CORRECT)
   → Card montre "En attente" + bouton "Connecter Amazon"

2. User clique "Connecter Amazon"
   → BFF /api/amazon/oauth/start
   → API compat /api/v1/marketplaces/amazon/oauth/start
   → Proxy vers keybuzz-backend
   → Backend genere Amazon auth URL (LWA)

3. User autorise sur Amazon
   → Amazon redirige vers platform-api callback

4. Backend callback (amazon.routes.js):
   a. completeAmazonOAuth() — echange code → tokens, stocke dans Vault
   b. ensureInboundConnection() — cree inbound_connections + inbound_addresses
   c. productDb.query(UPDATE tenant_channels SET status='active'...)
      ⚠ NE SET PAS inbound_email NI connection_ref ← CAUSE RACINE

5. BFF /api/amazon/status
   → API compat routes /api/v1/marketplaces/amazon/status
   → Query inbound_connections WHERE tenantId = ... AND status = 'READY'
   → Si READY → { connected: true }

6. Channels page:
   → Lit ch.status pour le badge ("Connecte" si active)
   → Lit ch.inbound_email pour afficher l'adresse
   → Lit ch.status pour le bouton ("pending" → bouton, "active" → "OAuth actif")

7. Orders page:
   → Appelle /api/amazon/status
   → Si connected: true → affiche marketplace + bouton orange
```

### Cause racine identifiee

**Le callback OAuth (amazon.routes.js lignes ~285-303) fait :**
```javascript
await productDb.query(
  `UPDATE tenant_channels SET status = 'active', activated_at = NOW(), connected_at = NOW()
   WHERE tenant_id = $1 AND provider = 'amazon' AND status != 'active'`,
  [tenantId]
);
```

**Mais NE set PAS :**
- `inbound_email` (l'adresse email generee dans inbound_addresses)
- `connection_ref` (l'ID de l'inbound_connections cree juste avant)

**Consequences :**
- Le channel passe a `active` mais sans email ni reference
- L'UI affiche "Connecte" / "OAuth actif" (se base sur status)
- L'email inbound n'apparait pas (inbound_email est NULL)
- Le status endpoint fonctionne (il lit inbound_connections, pas tenant_channels)
- Mais l'UI Channels et Orders sont incoherents

### Pourquoi les anciens tenants marchent

Pour ecomlg-001 et switaa-sasu-mnc1x4eq, l'email a ete propage par un processus
separe (probablement l'auto-provisioning via le status endpoint ou un script
d'activation historique). Le code actuel du callback ne le fait plus.

## 3. Chaine complete Channels / Orders / Inbound

### Channels page
- **Source donnees** : `fetchTenantChannels()` → BFF `/api/channels/list` → API `/channels?tenantId=...`
- **Badge** : `getStatusBadge(ch.status)` — "Connecte" si `active`, "En attente" si `pending`
- **Bouton** : `ch.provider === "amazon" && ch.status === "pending"` → "Connecter Amazon"
- **OAuth actif** : `ch.provider === "amazon" && ch.status === "active"` → texte vert
- **Email inbound** : `ch.inbound_email` — affiche seulement si non-null

### Orders page
- **Detection** : `detectMarketplaces()` → fetch `/api/amazon/status?tenant_id=...`
- **BFF** : `app/api/amazon/status/route.ts` → API `BACKEND_URL/api/v1/marketplaces/amazon/status`
- **API** : `compat/routes.ts` → query `inbound_connections WHERE "tenantId" = $1 AND status = 'READY'`
- **Bouton orange** : visible si `activeMarketplaces.length > 0`
- **Sync** : POST `/api/orders/sync-all`

### Status endpoint
- **Route** : GET `/api/v1/marketplaces/amazon/status` (enregistree dans compat/routes.ts)
- **Source** : table `inbound_connections` (NOT MarketplaceConnection)
- **Fallback** : recherche par prefixe tenant_id (PH-AMZ-ORDERS-LIST-SYNC-BUTTON-FIX-04)

## 5. Fix applique (3 corrections)

### FIX A — Client Channels page (`app/channels/page.tsx`)

Badge et bouton bases sur `connection_ref` au lieu de `status` seul :

```typescript
// Badge : "OAuth requis" si active sans connection_ref
const needsOAuth = ch.provider === "amazon" && ch.status === "active" && !ch.connection_ref;
const effectiveBadge = needsOAuth
  ? { label: "OAuth requis", cls: "bg-amber-..." }
  : badge;

// Bouton : montre "Connecter Amazon" si pas de connection_ref
{ch.provider === "amazon" && (ch.status === "pending" || !ch.connection_ref) && (
  <button>Connecter Amazon</button>
)}

// OAuth actif : seulement si connection_ref present
{ch.provider === "amazon" && ch.connection_ref && ch.status === "active" && (
  <span>OAuth actif</span>
)}
```

**Image** : `v3.5.249-ph145.4-amz-channel-fix-dev`

### FIX B — API status endpoint (`compat/routes.ts`)

Auto-provisioning : quand `inbound_connections` est READY, propage les emails
et connection_ref vers `tenant_channels` :

```typescript
// Quand inbound_connections READY trouve :
const addresses = await pool.query(
  `SELECT country, "emailAddress" FROM inbound_addresses WHERE "connectionId" = $1`,
  [connId]
);
for (const addr of addresses.rows) {
  await pool.query(
    `UPDATE tenant_channels
     SET inbound_email = $1, connection_ref = $2, status = 'active', updated_at = NOW()
     WHERE tenant_id = $3 AND provider = 'amazon' AND country_code = $4
       AND (inbound_email IS NULL OR connection_ref IS NULL)`,
    [addr.emailAddress, connId, tenantId, addr.country]
  );
}

// Fallback : check tenant_channels avec connection_ref comme source secondaire
const channelCheck = await pool.query(
  `SELECT COUNT(*) as cnt FROM tenant_channels
   WHERE tenant_id = $1 AND provider = 'amazon' AND status = 'active' AND connection_ref IS NOT NULL`,
  [tenantId]
);
```
