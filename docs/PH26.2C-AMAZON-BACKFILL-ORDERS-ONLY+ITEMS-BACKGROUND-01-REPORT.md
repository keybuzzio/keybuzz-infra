## Résumé

Le backfill Amazon a été optimisé en mode **ORDERS_ONLY** :
- Les orders sont importés **sans les items** (très rapide)
- Les items sont remplis en **background** par un worker séparé

---

## Architecture implémentée

### A) Backfill Fast: Orders ONLY

```
POST /api/v1/orders/backfill/start?tenantId=X&months=24
  ↓
Retourne 202 immédiatement
  ↓
Worker claim tenant (FOR UPDATE SKIP LOCKED)
  ↓
Pour chaque chunk de 30 jours:
  - Fetch orders SP-API (pagination, NextToken)
  - Bulk upsert 300 orders/transaction
  - NO items fetch
  ↓
Update progress → next chunk
```

### B) Items Fill: Background

```
POST /api/v1/orders/backfill/items/fill?tenantId=X
  ↓
Pour chaque order sans items (batch de 20):
  - Fetch items SP-API
  - Insert items
  - Rate limit strict: 30 calls/min/tenant
```

## Recommandations Production

### Rate Limiting SP-API

1. **Délai entre requêtes**: 500ms minimum (actuel: 200ms)
2. **Backoff on 429**: 5-10s avec exponential backoff
3. **Max retries**: 5 au lieu de 3
4. **Chunk size**: Réduire à 15-20 jours si rate limit trop agressif

### Workers K8s

| Worker | Replicas | Usage |
|--------|----------|-------|
| `orders-backfill-worker` | 2-3 | Multi-tenant parallel |
| `items-fill-worker` | 1 | Rate limit strict |

### CronJob recommandé

```yaml
# Items fill: toutes les 10 minutes
schedule: "*/10 * * * *"
# Orders backfill: toutes les 5 minutes (si PENDING/RUNNING)
schedule: "*/5 * * * *"
```
