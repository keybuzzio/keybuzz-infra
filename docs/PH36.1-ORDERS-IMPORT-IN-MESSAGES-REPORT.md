## 2. Cause Racine

### Problème : Octopia products vides + montant 0

L'import PH36.0 utilisait `octopiaOrder.items` ou `octopiaOrder.orderLines` pour extraire les produits. Or l'API Octopia v2 retourne les lignes de commande dans `octopiaOrder.lines`, avec :

- `line.offer.productTitle` → nom du produit
- `line.totalPrice.offerPrice` → prix
- `line.offer.sellerProductId` → SKU
- `line.offer.productGtin` → GTIN/EAN
- `line.quantity` → quantité
- `totalPrice.sellingPrice` → montant total
- `billingAddress.firstName` + `lastName` → client
- `lines[].parcels[].carrierName` → transporteur

### Schéma DB

`orders.external_order_id` est `TEXT` — compatible avec les OrderID alphanumériques Octopia. Aucun fix nécessaire.

### Cooldown / Retry

Le bouton "Réessayer l'import" appelle `performImport(false)` qui **ne passe PAS** par `canAutoImport()`. Le cooldown de 10 min ne bloque que l'auto-import à l'ouverture du panneau. Pas de fix nécessaire.

## 3. Fix appliqué

### Fichier modifié : `keybuzz-api/src/modules/orders/routes.ts`

#### 3a. Remplacement `importOctopiaOrderInline`

- Extraction de `lines` au lieu de `items`/`orderLines`
- Mapping `line.offer.productTitle`, `line.totalPrice.offerPrice`, etc.
- Extraction tracking depuis `lines[0].parcels[0]`
- Calcul `totalAmount` depuis `totalPrice.sellingPrice` / `offerPrice`
- Customer name depuis `billingAddress.firstName` + `lastName`
- Customer address depuis `billingAddress` (addressLine1, postalCode, city, countryCode)

#### 3b. Ajout auto-enrichissement Octopia dans `GET /:orderId`

Si `channel === 'octopia'` et `products` vides, le handler extrait les données depuis `raw_data.lines` (stocké en JSONB à l'import) et met à jour la DB. Cela assure que même les commandes importées avant ce fix s'enrichissent au premier accès.
