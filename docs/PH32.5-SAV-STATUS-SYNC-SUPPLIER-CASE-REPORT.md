## 1. Diagnostic — Où est stocké le Statut SAV ?

| Élément | Détail |
|---|---|
| **Table** | `conversations` |
| **Colonne** | `sav_status` (VARCHAR, nullable) |
| **Horodatage** | `sav_updated_at` (TIMESTAMPTZ) |
| **Valeurs existantes** | `to_process`, `waiting`, `in_progress`, `closed`, `NULL` |
| **Endpoint** | `PATCH /messages/conversations/:id/sav-status` |

Le champ `sav_status` est une **vraie colonne DB** sur `conversations`, mise à jour manuellement par l'agent via l'UI de la boîte de réception.

La table `supplier_cases` possède sa propre colonne `status` (type `supplier_case_status` enum) avec les valeurs : `pending_supplier`, `awaiting_customer`, `in_progress`, `closed`.

**Avant PH32.5** : ces deux statuts étaient **complètement indépendants** — aucune synchronisation.

---

## 2. Implémentation — Sync Backend

### Fichier modifié

`keybuzz-api/src/modules/suppliers/suppliers.routes.ts`

### Fonction helper

```typescript
const SAV_STATUS_MAP: Record<string, string> = {
  pending_supplier: 'waiting',
  in_progress: 'in_progress',
  awaiting_customer: 'waiting',
  closed: 'closed',
};

async function syncSavStatus(pool: any, conversationId: string, supplierCaseStatus: string): Promise<void> {
  const savStatus = SAV_STATUS_MAP[supplierCaseStatus];
  if (!savStatus || !conversationId) return;
  await pool.query(
    `UPDATE conversations SET sav_status = $1, sav_updated_at = NOW() WHERE id = $2`,
    [savStatus, conversationId]
  );
  console.log(`[PH32.5] Synced sav_status=${savStatus} for conversation=${conversationId}`);
}
```

### Mapping V1

| supplier_case.status | → conversations.sav_status | Badge UI (existant) |
|---|---|---|
| `pending_supplier` | `waiting` | Orange — "En attente" |
| `in_progress` | `in_progress` | Bleu — "En cours" |
| `awaiting_customer` | `waiting` | Orange — "En attente" |
| `closed` | `closed` | Vert — "Clos" |

> Les valeurs `sav_status` utilisées sont celles **déjà supportées** par le client, permettant un patch **backend-only** sans aucun rebuild client (zéro risque de régression).

### 4 Points de synchronisation

| # | Trigger | Ligne(s) | Description |
|---|---|---|---|
| 1 | `POST /supplier-cases` (create) | ~305, ~325 | Sync à la création d'un dossier SAV |
| 2 | `PATCH /supplier-cases/:id/status` | ~362 | Sync au changement manuel de statut |
| 3 | `POST /supplier-inbound` | ~617-624 | Sync à la réception d'une réponse fournisseur |
| 4 | `POST /supplier-cases/:id/send-email` | ~826-833 | Sync après envoi d'email au fournisseur |

Tous les points sont **idempotents** — la même opération peut être rejouée sans effet de bord.
