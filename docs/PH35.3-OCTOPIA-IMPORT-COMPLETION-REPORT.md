# PH35.3 — Octopia Import Completion & Sync (DEV only)

## Objectif

Garantir que 100% des discussions Octopia actives soient visibles dans KeyBuzz.

## Analyse de l'API Octopia

### Pagination

| Parametre | Valeur |
|---|---|
| `pageSize` | max testee : 50 (100 → HTTP 400) |
| `pageIndex` | pagination via ce parametre (1-indexed) |
| `Link` header | contient `rel="next"`, `rel="first"`, `rel="last"` |
| Response | `{ items: [...], itemsPerPage: N }` |

### Comptage

| Metrique | Valeur |
|---|---|
| Discussions Octopia (API) | **232** |
| Pages (pageSize=50) | 5 |
| Conversations KeyBuzz avant PH35.3 | 10 |
| Conversations KeyBuzz apres backfill | **232** |

## Modifications

### 1. `octopiaImport.service.ts` — Refactored + Backfill + Sync

- **`importSingleDiscussion()`** : logique d'import extraite en fonction reutilisable
- **`backfillOctopiaDiscussions()`** : pagination complete avec `pageSize=50`, boucle `pageIndex`, arret quand `items.length < PAGE_SIZE`
- **`syncOctopiaDiscussions()`** : sync incremental, utilise `updatedSince` = dernier `updated_at` en DB, met a jour les statuts des conversations existantes

### 2. `octopia.routes.ts` — Nouvelles routes

- `POST /marketplaces/octopia/backfill` : backfill complet, parametres `{ tenantId, fromDate? }`
- `POST /marketplaces/octopia/sync` : sync incremental, parametres `{ tenantId, updatedSince? }`

### 3. `octopiaSyncWorker.ts` — Worker periodique

- Pattern identique a `slaBatchWorker.ts`
- Intervalle : 10 minutes
- Delai initial : 30 secondes
- Parcourt tous les tenants Octopia actifs
- Appelle `syncOctopiaDiscussions()` pour chaque

### 4. `app.ts` — Registration du worker

- Import et appel de `startOctopiaSyncWorker()` au demarrage
