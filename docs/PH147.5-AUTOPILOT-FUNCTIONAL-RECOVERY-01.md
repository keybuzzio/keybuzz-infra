## ETAPE 2 — REPRODUCTION DES BUGS

### Bug A : BFF routes sans auth headers

- **Reproduction** : Inspection du code source des 5 routes BFF autopilot
- **Constat** : Toutes envoyaient `Cookie: request.headers.get('cookie')` au lieu de `X-User-Email` / `X-Tenant-Id`
- **Impact** : L'API backend recevait les requetes sans identification utilisateur/tenant

### Bug B : Autopilot `evaluate` retourne `MODE_NOT_AUTOPILOT`

- **Reproduction** : `POST /autopilot/evaluate` pour tenant `ecomlg-001` (plan PRO)
- **Constat** : `PLAN_CAPABILITIES.PRO.maxMode = 'suggestion'` → mode plafonné à suggestion
- **Impact** : **Normal et attendu** — PRO = suggestions seules, AUTOPILOT = drafts auto

---

## ETAPE 3 — DIAGNOSTIC PRECIS

### Bug A — BFF Auth Headers

**Cause racine** : Les 5 routes BFF autopilot ont ete codees sans extraction de la session NextAuth. Elles copient le pattern initial (Cookie forwarding) au lieu du pattern correct utilise par `/api/ai/assist` (session extraction + X-User-Email + X-Tenant-Id).

**Ce qui est present** :

- Backend API complet et fonctionnel
- Routes BFF existent avec la bonne structure de fichiers
- UI client (InboxTripane) appelle `/api/autopilot/draft` correctement

**Ce qui manquait** :

- `import { getServerSession } from 'next-auth'` dans les 5 routes BFF
- `import { authOptions } from '../../auth/[...nextauth]/auth-options'`
- Extraction `session.user.email` et forwarding via `X-User-Email` + `X-Tenant-Id`

### Bug B — Plan gating Autopilot

**Condition attendue** : Autopilot genere des drafts automatiquement
**Condition reelle** : Le tenant `ecomlg-001` est sur plan `PRO` → `maxMode: 'suggestion'`
**Ce n'est PAS un bug** : C'est le comportement attendu de la matrice `PLAN_CAPABILITIES`

Le tenant `switaa-sasu-mnc1x4eq` (plan `AUTOPILOT`) a 46 `autopilot_reply` et 22 `autopilot_escalate` en DB, confirmant que l'engine fonctionne pour les plans AUTOPILOT.
