## 2. PLAN iptables (H2)

### 2.1 Contexte reseau


| Reseau            | CIDR                            | Usage                        |
| ----------------- | ------------------------------- | ---------------------------- |
| Hetzner prive     | `10.0.0.0/16`                   | Communication inter-serveurs |
| K8s pods (Calico) | `10.244.0.0/16`                 | Trafic pods                  |
| K8s services      | `10.96.0.0/12`                  | ClusterIP services           |
| Ingress NodePorts | `31169` (HTTP), `31631` (HTTPS) | Trafic web entrant           |


### 2.2 Regles proposees pour les serveurs DATA (Vault, PG, Redis, RabbitMQ, MinIO)

```bash
#!/bin/bash
# iptables-hardening-data.sh
# A appliquer sur : 10.0.0.120-122 (PG), 10.0.0.123-125 (Redis), 
#                   10.0.0.126-128 (RabbitMQ), 10.0.0.134 (MinIO), 10.0.0.150 (Vault)

# Flush existant
iptables -F INPUT

# Loopback
iptables -A INPUT -i lo -j ACCEPT

# Connexions etablies
iptables -A INPUT -m state --state ESTABLISHED,RELATED -j ACCEPT

# SSH depuis reseau prive + bastion
iptables -A INPUT -s 10.0.0.0/16 -p tcp --dport 22 -j ACCEPT
iptables -A INPUT -s 46.62.171.61/32 -p tcp --dport 22 -j ACCEPT

# Reseau prive Hetzner (inter-services)
iptables -A INPUT -s 10.0.0.0/16 -j ACCEPT

# K8s pods (Calico) — necessaire pour que les pods accedent aux services
iptables -A INPUT -s 10.244.0.0/16 -j ACCEPT

# K8s services (ClusterIP)
iptables -A INPUT -s 10.96.0.0/12 -j ACCEPT

# ICMP (ping diagnostic)
iptables -A INPUT -p icmp -j ACCEPT

# Drop tout le reste
iptables -A INPUT -j DROP

# Persister
iptables-save > /etc/iptables/rules.v4
```

### 2.3 Regles pour les K8s workers (NE PAS appliquer les regles data)

Les K8s workers (10.0.0.110-114) ont besoin de NodePorts ouverts pour le trafic Ingress. Ils ont deja des regles iptables complexes gerees par Calico/kube-proxy. **NE PAS toucher aux iptables des workers K8s.**

### 2.4 Regles pour les K8s masters (NE PAS toucher)

Les masters (10.0.0.100-102) gerent l'API server (port 6443) et les communications cluster. **NE PAS toucher.**

### 2.5 Procedure d'application

```
1. SCP le script sur le serveur cible
2. SSH au serveur
3. Executer le script
4. Tester immediatement :
   - SSH fonctionne encore (depuis le bastion)
   - Le service local repond (depuis le bastion via IP privee)
   - Un pod K8s peut acceder au service
5. Si echec : reboot du serveur (iptables non persistees sans iptables-save)
```

## 3. PLAN VAULT TOKEN (H3)

### 3.1 Etat actuel


| Element                   | Valeur                                                                                                  |
| ------------------------- | ------------------------------------------------------------------------------------------------------- |
| Secret `vault-root-token` | Present dans `keybuzz-api-dev`, `keybuzz-api-prod`                                                      |
| Secret `vault-app-token`  | Present dans `keybuzz-api-dev`, `keybuzz-api-prod`, `keybuzz-backend-dev`, `keybuzz-backend-prod` (43j) |
| ClusterSecretStore        | Utilise l'auth Kubernetes (pas le root token)                                                           |
| ExternalSecrets           | 30/30 synced — utilisent deja l'auth Kubernetes                                                         |


### 3.2 Constats

Les **ExternalSecrets** (30/30) utilisent deja l'authentification Kubernetes via les ClusterSecretStores, **PAS le root token**. C'est la methode recommandee.

Le `vault-root-token` est reference dans l'env `VAULT_TOKEN` du deployment API. Il est utilise par le code applicatif (helper `vaultFetch`) pour des lectures Vault directes (credentials Amazon, etc.).

Le `vault-app-token` existe deja (cree il y a 43 jours) avec un token `hvs.CAESI...` distinct du root.

### 3.3 Plan de migration


| Etape | Action                                                                                                         | Risque                          |
| ----- | -------------------------------------------------------------------------------------------------------------- | ------------------------------- |
| 1     | Verifier que `vault-app-token` a les policies necessaires (lecture `secret/`*, `keybuzz/*`)                    | Aucun (lecture seule)           |
| 2     | Modifier le deployment API DEV pour pointer `VAULT_TOKEN` vers `vault-app-token` au lieu de `vault-root-token` | Faible (DEV d'abord)            |
| 3     | Valider que toutes les lectures Vault fonctionnent en DEV (credentials Amazon, JWT, etc.)                      | Aucun                           |
| 4     | Appliquer la meme modification en PROD                                                                         | Faible (apres validation DEV)   |
| 5     | Revoquer le root token apres validation (optionnel, conserver en coffre physique)                              | Moyen — irrecuperable si besoin |


### 3.4 Rollback

Si le `vault-app-token` n'a pas les bonnes permissions :

```bash
kubectl set env deployment/keybuzz-api VAULT_TOKEN=<root_token_value> -n keybuzz-api-prod
```

## 5. PLAN BIND (H4, H5)

### 5.1 Vault — Binder sur IP privee (H4)


| Fichier                  | Modification                                               |
| ------------------------ | ---------------------------------------------------------- |
| `/etc/vault.d/vault.hcl` | `address = "0.0.0.0:8200"` → `address = "10.0.0.150:8200"` |


**Impact** : le service K8s `vault` dans `default` pointe deja vers `10.0.0.150:8200`. Aucun impact fonctionnel.

**Procedure** :

```bash
ssh 10.0.0.150
sed -i 's/address     = "0.0.0.0:8200"/address     = "10.0.0.150:8200"/' /etc/vault.d/vault.hcl
systemctl restart vault
# Verifier : curl -s http://10.0.0.150:8200/v1/sys/health
# Verifier depuis K8s : kubectl run test --rm -it --restart=Never --image=curlimages/curl -- curl http://vault.default.svc.cluster.local:8200/v1/sys/health
```

**Rollback** : reverter le sed et restart.

### 5.2 MinIO — Binder sur IP privee (H5)


| Fichier                         | Modification probable                                                 |
| ------------------------------- | --------------------------------------------------------------------- |
| Service systemd ou config MinIO | Ajouter `--address 10.0.0.134:9000 --console-address 10.0.0.134:9001` |


**Procedure** :

```bash
ssh 10.0.0.134
# Identifier la config : systemctl cat minio
# Modifier MINIO_OPTS ou ExecStart pour binder sur 10.0.0.134
systemctl restart minio
# Verifier : curl -s http://10.0.0.134:9000/minio/health/live
```

**Rollback** : reverter la config et restart.
