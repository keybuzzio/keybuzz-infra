# PH34.1C — Trusted Internal Proxy Auth (Remove KEYBUZZ_DEV_MODE in PROD)

## Résumé

Remplacement du mécanisme `KEYBUZZ_DEV_MODE=true` en PROD par un système d'authentification trusted proxy basé sur un token interne partagé. Zero hardcode tenant.

## Cause racine du problème initial

Le middleware `devAuthMiddleware.ts` de `keybuzz-backend` n'acceptait `X-User-Email` que si `KEYBUZZ_DEV_MODE=true`. En PH34.1B, cette variable avait été activée en PROD comme workaround. PH34.1C remplace ce workaround par un mécanisme propre.

## Architecture trusted proxy

```
[Navigateur] → [NextAuth session] → [BFF Next.js] → [keybuzz-backend]
                                        │                    │
                                        │  X-User-Email      │
                                        │  X-Tenant-Id       │  devAuthMiddleware
                                        │  X-Internal-Token  │  vérifie le token
                                        └────────────────────┘
```

### Flux d'authentification

1. L'utilisateur est authentifié via NextAuth (session cookie)
2. Le BFF extrait l'email de la session (`getServerSession`)
3. Le BFF envoie au backend :
   - `X-User-Email` : email vérifié par NextAuth
   - `X-Tenant-Id` : tenant courant
   - `X-Internal-Token` : token partagé via K8s secret
4. Le backend vérifie :
   - **Path 1 (Proxy)** : Si `X-Internal-Token` valide → accepte, même si l'utilisateur n'existe pas dans la DB backend (contexte créé depuis les headers)
   - **Path 2 (DEV)** : Si `KEYBUZZ_DEV_MODE=true` → accepte `X-User-Email` (DEV uniquement)
   - **Path 3 (JWT)** : Sinon → authentification JWT classique

## Implémentation

### 1. Secret K8s partagé

Secret `keybuzz-internal-proxy` créé dans les 4 namespaces :
- `keybuzz-backend-dev`
- `keybuzz-backend-prod`
- `keybuzz-client-dev`
- `keybuzz-client-prod`

Token : 64 caractères hex (généré via `openssl rand -hex 32`).

### 2. Backend middleware (`devAuthMiddleware.ts`)

Modifié pour accepter 3 chemins d'auth :
- **Proxy** : `X-Internal-Token` valide + `X-User-Email` → contexte depuis headers (pas de lookup DB requis)
- **DEV** : `KEYBUZZ_DEV_MODE=true` + `X-User-Email` → lookup DB requis
- **JWT** : `Authorization: Bearer <token>` → vérification JWT standard

### 3. BFF routes patchées (6 fichiers)

Toutes les routes Amazon BFF envoient désormais `X-Internal-Token: INTERNAL_PROXY_TOKEN` :

| Route | Status |
|-------|--------|
| `amazon/disconnect/route.ts` | Patché |
| `amazon/inbound-address/route.ts` | Patché |
| `amazon/inbound-address/send-validation/route.ts` | Patché |
| `amazon/oauth/start/route.ts` | Patché |
| `amazon/status/route.ts` | Patché |
| `debug-amazon-connect/route.ts` | Patché |

### 4. Configuration env

| Env | Variable | Valeur |
|-----|----------|--------|
| PROD backend | `KEYBUZZ_DEV_MODE` | **`false`** |
| PROD backend | `KEYBUZZ_INTERNAL_PROXY_TOKEN` | from secret |
| PROD client | `KEYBUZZ_INTERNAL_PROXY_TOKEN` | from secret |
| DEV backend | `KEYBUZZ_DEV_MODE` | `true` |
| DEV backend | `KEYBUZZ_INTERNAL_PROXY_TOKEN` | from secret |
| DEV client | `KEYBUZZ_INTERNAL_PROXY_TOKEN` | from secret |

## Validations

### Tests serveur

| Test | Résultat |
|------|----------|
| User inconnu + proxy token + tenantId | **200** (contexte créé depuis headers) |
| User inconnu sans token (PROD, DEV_MODE=false) | **401** (correctement rejeté) |
| User connu + proxy token | **200** (lookup DB réussi) |
| Cross-namespace (client → backend PROD) | **200** |
| `KEYBUZZ_DEV_MODE` en PROD | **false** |

### Sécurité

- Le token n'est jamais exposé publiquement (K8s secret → env var serveur uniquement)
- La communication BFF→backend est interne au cluster K8s (pas exposée via ingress)
- Un token invalide produit un log `[ProxyAuth] Invalid internal token attempt` et fall-through vers JWT (401)
- Aucun hardcode de tenantId ou email dans le code
