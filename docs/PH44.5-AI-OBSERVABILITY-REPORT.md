# PH44.5 — AI Decision Debug & Observability Layer

> Date : 8 mars 2026
> Deploye : DEV uniquement (v3.5.51-ph445-observability-dev)
> Rollback : v3.5.50-ph44-tenant-policy-dev

---

## 1. Objectif

Ajouter un systeme d'observabilite IA permettant de comprendre exactement :
- Pourquoi l'IA a repondu de telle facon
- Quelles regles ont ete appliquees
- Quel historique a ete utilise
- Quel contexte commande/fournisseur a ete injecte

**Sans modifier le comportement IA actuel.**

---

## 2. Composants implementes

### 2.1 Endpoint GET /ai/policy/effective

Nouvel endpoint de debug qui expose la politique reellement appliquee pour une conversation.

**Route** : `GET /ai/policy/effective?tenantId=xxx&conversationId=yyy`
**Fichier** : `src/modules/ai/ai-policy-debug-routes.ts`
**Enregistrement** : `app.register(aiPolicyDebugRoutes, { prefix: '/ai' })`

**Securite** :
- Header `X-User-Email` obligatoire (401 si absent)
- Verification `user_tenants` (403 si l'utilisateur n'appartient pas au tenant)
- Conversation doit exister pour le tenant (404 sinon)

### 2.2 Decision Context dans ai-assist-routes.ts

Apres chaque appel IA reussi, un objet `decisionContext` est capture et :
1. Inclus dans la reponse API sous `response.decisionContext`
2. Persiste dans `ai_actions_ledger.decision_context` (UPDATE par `request_id`)
3. Logue dans `ai_action_log` avec `action_type = 'AI_DECISION_TRACE'`

**Structure du decisionContext** :
```json
{
  "policyLayers": {
    "globalPolicy": true,
    "tenantPolicy": false,
    "historicalContext": true,
    "orderContext": true,
    "supplierContext": false
  },
  "savScenario": "damaged_product",
  "savConfidence": 0.46,
  "historicalMatch": false,
  "historicalCasesCount": 0,
  "dominantResolution": null,
  "antiPatterns": ["photo_before_decision"],
  "tenantPolicyActive": false,
  "kbSource": "inbox_contextualized",
  "model": "kbz-standard"
}
```

### 2.3 Migration DB

```sql
ALTER TABLE ai_actions_ledger ADD COLUMN IF NOT EXISTS decision_context JSONB;
```

Colonnes `ai_actions_ledger` apres migration :
`id, tenant_id, delta, reason, request_id, conversation_id, created_at, kb_actions, cost_usd, decision_context`

### 2.4 Integration PH44 (Tenant Policy)

L'integration PH44 (tenant policy loader) a ete consolidee dans `ai-assist-routes.ts` :
- Import : `loadTenantPolicy`, `buildTenantPolicyBlock`, `TenantAiPolicy`
- Injection dans `buildSystemPrompt` entre SAV policy et historical block
- Metadata dans la reponse sous `response.tenantPolicy`

## 5. Ordre des blocs prompt IA (actualise)

```
1. GLOBAL_POLICY (base prompt, regles generales)
2. SAV_POLICY (PH41 — classification scenario)
3. TENANT_POLICY (PH44 — politique tenant configurable)
4. HISTORICAL_CONTEXT (PH43 — cas similaires + anti-patterns)
5. ORDER_CONTEXT (commande liee)
6. TENANT_RULES (regles ai_rules du vendeur)
7. SUPPLIER_CONTEXT (fournisseur lie)
```
