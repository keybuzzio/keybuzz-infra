## ANALYSE CAUSE RACINE DES ECHECS PRECEDENTS

### PH154.x : branche consolidation manquait des composants PH151

La branche `consolidation/client-v3.5.63-inbox-m1-candidate` est **privee de 6 composants** presents dans `ph152.6-client-parity` :

- AICaseSummary.tsx, ConversationSummaryBar.tsx, MessageBubble.tsx
- MessageFilterToggle.tsx, messageClassifier.ts, AutopilotConversationFeedback.tsx

**MAIS** : verification du chunk compile de l'image deployee montre que ces composants ne sont PAS non plus dans l'image stable actuelle. Les captures PROD montrent des features (DEMANDE/ACTIONS/ETAT) qui proviennent d'un deploiement anterieur.

### PH154.1.1 : badge "Suivi de livraison" pour TOUTES les commandes

La condition du badge type etait :

```
order.trackingCode && order.deliveryStatus !== "PENDING"
```

→ Vrai pour ~90% des commandes → badge afiche partout → "ligne parasite".

### PH154.1.2 : correction = detection par sujet

Le badge type ne s'affiche QUE si le contexte le justifie :

- `isSystemNotification === true` → Bell + "Notification Amazon"
- `conversationType === "RETURN_REQUEST"` → Package + "Demande de retour"
- Sujet contient mots-cles livraison/colis/tracking → Truck + "Suivi de livraison"
- Sinon → **RIEN** (pas de badge, pas de bordure, pas de ligne)

## LOGIQUE DE CLASSIFICATION (inchangee, conservative)

`isAmazonAutoMessage()` : exige **>=2 signaux forts** parmi 10 (copyright+amazon, seller central, preferences notification, ne pas repondre, etc.)

Un message client comme "JE DEMANDE LE REMBOURSEMENT" = 0 signaux → classe Client.
