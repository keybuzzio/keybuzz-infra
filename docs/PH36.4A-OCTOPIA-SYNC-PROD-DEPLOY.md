# PH36.4A — Octopia Discussions Sync — DEV + PROD

**Date** : 22 février 2026
**Auteur** : Agent Cursor
**Scope** : DEV + PROD
**Status** : DEPLOYED

---

## Résumé

Activation du worker de synchronisation incrémentale des discussions Octopia, à la fois en DEV et en PROD. Le worker importe automatiquement les nouvelles discussions et messages depuis l'API Octopia toutes les 10 minutes.

## Cause racine

L'image API déployée ne contenait pas :
- `/app/dist/modules/marketplaces/octopia/` (module d'import Octopia)
- `/app/dist/workers/octopiaSyncWorker.js` (worker de sync périodique)
- L'appel `startOctopiaSyncWorker()` dans `app.js`

Le code source existait sur le serveur mais n'avait jamais été compilé dans les images Docker déployées.

## Approche : Patch chirurgical

Un rebuild complet depuis le source serveur a été tenté initialement pour DEV, mais il a introduit des régressions : le source serveur avait divergé de l'image en production (routes `compatRoutes`, `ordersRoutes` manquantes dans le nouveau source).

**Solution retenue** : patch chirurgical sur l'image existante stable :
1. Extraction des fichiers octopia compilés depuis une image intermédiaire
2. Injection dans l'image de base (préservant toutes les routes existantes)
3. Patch minimal de `app.js` (import + start du worker)

## Fix additionnel : Amazon sync error handling

Le bouton "Synchroniser Amazon" produisait une erreur persistante ("Identifiants Amazon non configurés") qui polluait le state de sync global. Le bouton Octopia affichait ensuite la même erreur Amazon car la page poll `/api/orders/sync-status` au chargement.

**Cause** : `POST /sync-all` lançait le sync async (retour 200), puis l'async échouait et stockait l'erreur en mémoire. Le polling renvoyait cette erreur indéfiniment.

**Fix** : Ajout d'un **pre-check des credentials Amazon** dans le handler `/sync-all`, AVANT le lancement async. Si les credentials sont absents, retour HTTP 400 immédiat sans pollution du state.

```
Avant: sync-status => {"status":"error","error":"Identifiants Amazon non configurés..."}
Après: sync-status => {"status":"idle","message":"Aucune synchronisation en cours"}
       sync-all    => 400 {"status":"not_configured","error":"...","message":"Veuillez configurer..."}
```

## Leçons apprises

1. **Ne jamais rebuild depuis le source serveur** quand celui-ci a divergé de l'image en production. Toujours partir de l'image stable existante.
2. **Patch chirurgical** = ajouter uniquement les fichiers nécessaires + patch minimal de app.js.
3. **Vérifier les conflits de routes** : en PROD, `compat` fournit déjà des routes `/marketplaces/octopia/*`, donc l'enregistrement des routes réelles crée un `FST_ERR_DUPLICATED_ROUTE`.
4. **Ne jamais stocker d'erreur en state mémoire avant de valider les pré-conditions** : un sync async qui échoue après retour 200 pollue le state de façon persistante.

## Comportement du worker

- **Intervalle** : 10 minutes
- **Mode** : incrémental (`updatedSince = MAX(updated_at)` depuis la DB)
- **Sécurité** : advisory lock PostgreSQL par tenant (pas de doublons)
- **Tenants** : traite tous les tenants avec une connexion Octopia active
