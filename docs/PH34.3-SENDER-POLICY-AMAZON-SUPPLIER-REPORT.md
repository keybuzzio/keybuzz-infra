## Probleme
Le outbound worker ne lisait pas `fromAddress` ni `replyToAddress` depuis `delivery_trace` JSON.
La table `outbound_deliveries` n'a pas de colonnes `reply_to`, `from_address`, `subject`.
Resultat : tous les emails SMTP (supplier) partaient avec `From: noreply@keybuzz.io` et sans Reply-To.

## Cause racine
Le code supplier (`suppliers.routes.ts`) stocke correctement `fromAddress: support@keybuzz.io` et 
`replyToAddress: sav.<tenantId>.<caseId>@inbound.keybuzz.io` dans `delivery_trace` JSON. 
Mais le worker `outboundWorker.js` (handler `email_forward/smtp`) ne lisait que les colonnes 
de la table (qui n'existent pas), pas le JSON trace.

## Fix
Patch du handler `email_forward/smtp` dans `outboundWorker.js` pour extraire depuis `delivery_trace` :
```javascript
// PH34.3: Extract from/replyTo from delivery_trace
const dt = typeof delivery.delivery_trace === 'string' 
  ? JSON.parse(delivery.delivery_trace) : delivery.delivery_trace;
const traceFrom = dt?.fromAddress || null;
const traceReplyTo = dt?.replyToAddress || null;
const traceSubject = dt?.emailSubject || null;

await sendEmail({
  to: delivery.target_address,
  from: traceFrom || undefined,          // support@keybuzz.io pour supplier
  subject: traceSubject || ...,
  replyTo: traceReplyTo || undefined,     // sav.<tenantId>.<caseId>@inbound
  ...
});
```

## Politique Sender finale

| Type | From | Reply-To | Guard |
|------|------|----------|-------|
| **Amazon Marketplace** | `amazon.<tenantId>.<country>.<token>@inbound.keybuzz.io` | `delivery.reply_to` | `validateAmazonOutboundConfig` (bloque si pas VALIDATED) |
| **Supplier Contact** | `support@keybuzz.io` (via `delivery_trace.fromAddress`) | `sav.<tenantId>.<caseId>@inbound.keybuzz.io` (via `delivery_trace.replyToAddress`) | Aucun (fallback noreply) |
| **Email Forward** | `delivery_trace.fromAddress` ou `noreply@keybuzz.io` | `delivery_trace.replyToAddress` ou null | Aucun |
| **OTP / Magic Link** | `"KeyBuzz" <noreply@keybuzz.io>` | Aucun | N/A (client-side) |
