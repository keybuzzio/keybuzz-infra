# PH35.1 — Octopia Adapter Layer (Lecture seule) — DEV

## Entite choisie : Discussions Octopia

Les **discussions** Octopia sont l'equivalent des "dossiers clients" : elles contiennent un sujet,
une reference commande, un statut (Open/Treated/Closed), et des messages echanges.

### Endpoint Octopia utilise

| Endpoint | Methode | Description |
|----------|---------|-------------|
| `GET /discussions?pageSize=N` | GET | Liste des discussions (avec header `sellerId`) |
| `GET /discussions/{id}` | GET | Detail d'une discussion + messages[] |

### Payload minimal d'une discussion

```json
{
  "discussionId": "69849ce4ad2f28a5e9d2a6b2",
  "subject": "Notification d'exception UPS...",
  "orderReference": "2601241824KLYU1",
  "isOpen": true,
  "status": "Treated",
  "typologyCode": "Order",
  "customerId": "000000093L9F",
  "sellerId": "273340",
  "salesChannel": "CDISFR",
  "createdAt": "2026-02-05T13:35:16.42Z",
  "updatedAt": "2026-02-18T14:24:14.64Z",
  "messages": [
    { "messageId": "...", "body": "contenu...", "discussionId": "..." }
  ]
}
```

## Mapping Octopia → KeyBuzz

| Octopia | conversations | Valeur |
|---------|---------------|--------|
| discussionId | thread_key | `octopia:{discussionId}` (deduplication) |
| subject | subject | Tronque a 200 chars |
| orderReference | order_ref | Ref commande Octopia |
| customerId | customer_name / customer_handle | "Client {customerId}" |
| status + isOpen | status | ToTreat→pending, Treated+open→open, !open→resolved |
| - | channel | `'octopia'` |
| - | priority | `'normal'` |
| - | sla_state | `'OK'` |

### Messages

- 1er message : resume structure (type, commande, date, statut, contenu original)
- Messages suivants : corps nettoye (HTML strip, CRLF normalise)
- Tous en `direction='inbound'`, `message_source='SYSTEM'`
- `metadata.source = 'octopia_import'`
