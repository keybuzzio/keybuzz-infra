# PH33.16D — Exemption paywall pour comptes internes (DEV + PROD)

## Objectif
Empecher le verrouillage paywall sur les tenants internes (eComLG) sans impacter les clients.

## Implementation

### Probleme rencontre
La table `tenants` est owned par `vault_admin` (superuser PostgreSQL). Aucun user disponible (API, Vault admin, Vault migrator) n'a les privileges `ALTER TABLE`. Le serveur DB (`10.0.0.10`) n'est pas accessible en SSH.

### Solution : table separee `tenant_billing_exempt`
Au lieu de modifier la table `tenants`, creation d'une table independante :

```sql
CREATE TABLE IF NOT EXISTS tenant_billing_exempt (
  tenant_id TEXT PRIMARY KEY,
  exempt BOOLEAN DEFAULT true,
  reason TEXT DEFAULT 'internal',
  created_at TIMESTAMPTZ DEFAULT NOW()
);
```

Le role Vault admin a les privileges `CREATE TABLE`, donc cette approche fonctionne.

### Marquage du tenant interne (sans hardcode de tenantId)
```sql
INSERT INTO tenant_billing_exempt (tenant_id, exempt, reason)
SELECT ut.tenant_id, true, 'internal_admin'
FROM user_tenants ut
JOIN users u ON u.id = ut.user_id
WHERE u.email = 'ludo.gonthier@gmail.com' AND ut.role = 'owner'
ON CONFLICT (tenant_id) DO UPDATE SET exempt = true, reason = 'internal_admin';
```

### Entitlement logic
Dans `getTenantEntitlement(tenantId)` :
1. Lookup `tenant_billing_exempt` via `isTenantExempt(pool, tenantId)`
2. Si `exempt = true` → `isLocked=false`, `lockReason='NONE'`, `billingExempt=true`
3. Les donnees subscription sont chargees pour info (non bloquantes)
4. Nouveau champ `billingExempt: boolean` dans la reponse
